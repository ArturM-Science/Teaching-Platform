# Lesson 3 · State & Session Management

An agent conversation has memory: turn five depends on turns one through four. But the server that handles turn five might not be the same machine that handled turn one. This lesson is about where conversation state lives so that the serving layer can stay stateless — the property that lets you run more than one copy of your service.

## Why the server must stay stateless

"Stateless" doesn't mean the conversation has no memory. It means **the server process holds no memory between requests** — every request carries or fetches everything it needs. The reason is scaling: production runs multiple replicas of your service behind a load balancer, and any replica might handle any request.

```
                    ┌─────────────┐
   request 1  ─────▶│  replica A  │   (holds messages in a global)
                    └─────────────┘
                    ┌─────────────┐
   request 2  ─────▶│  replica B  │   ← never saw request 1's messages
                    └─────────────┘
```

If replica A stored the conversation in a module-level variable, replica B has no idea it exists. The user's second message lands on B and the agent has amnesia. This is the single most common deployment bug for chat agents, and it's invisible in development because your laptop runs exactly one replica.

<Callout type="warning">
"It works locally" is the trap. One replica makes a stateful server look correct. The bug appears only under the thing you can't easily reproduce on a laptop: a second replica. Assume from the first line of code that another copy of your process exists and shares nothing with you.
</Callout>

## Where conversation state can live

If not in the server, then where? There are three honest answers, each with a different trade-off.

**1. In the client.** The browser (or calling service) holds the history and sends it with every request — exactly the `history: list[dict]` field from Lesson 1.

```python
class ChatRequest(BaseModel):
    message: str
    history: list[dict] = []   # client owns the conversation, replays it each turn
```

Simple and perfectly stateless — the server stores nothing. The downsides: the payload grows every turn (a long chat means a large request), and you trust the client not to tamper with history. Good for short conversations and prototypes.

**2. In a shared store, keyed by session id.** The client holds only a `session_id`; the server reads and writes the history in a database or cache that all replicas share.

```python
class ChatRequest(BaseModel):
    message: str
    session_id: str            # client holds only the key

@app.post("/chat")
def chat(req: ChatRequest):
    history = store.get(req.session_id)        # any replica can read it
    answer, history = run_agent(req.message, history, api_key)
    store.put(req.session_id, history)         # any replica can write it
    return {"answer": answer, "session_id": req.session_id}
```

The server is still stateless — it owns no memory of its own; it just borrows from the store. This is the standard production pattern. The history can grow without bloating requests, and the data outlives any single replica. The cost is the store itself: a Postgres table or a Redis instance, plus the latency of a read and write per turn.

**3. Split: durable history in a database, hot context in a cache.** The full transcript lives in Postgres (durable, queryable); the recent window the agent actually needs each turn lives in Redis (fast). This is what high-traffic systems do, and it connects directly to the memory architecture from Module 3 — short-term working memory versus long-term storage, now as an infrastructure decision.

<Callout type="key">
The server stays stateless; the *state* moves to a store every replica can reach. Client-held history is the simplest stateless option; a session-keyed shared store is the production default. Either way, no conversation lives inside a single server process — because that process can be replaced, restarted, or joined by a sibling at any moment.
</Callout>

## Session ids, identity, and cleanup

A session is just a key the client carries — usually a UUID minted on the first request and returned to the client to send back. Two practical rules:

- **Don't put anything secret in the session id itself.** It's an opaque lookup key, not an auth token. Authorize the *user* separately (the auth from Module 3's deployment story), then check that the user owns the session.
- **Expire sessions.** Conversation state accumulates forever if you let it. Set a TTL in the cache and an archival/cleanup policy in the database, or your store grows without bound and your costs follow.
