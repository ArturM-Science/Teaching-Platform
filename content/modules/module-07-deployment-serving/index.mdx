# Module 7 · Deployment & Serving

**Part 4 — Production** · 5 lessons + lab · ~6 hours total

<LearningObjectives>
- Explain why a notebook agent is not a product, and draw the request/response boundary that turns agent logic into a service
- Choose between synchronous, streaming (SSE), and async/queue serving patterns based on the task's latency profile
- Keep the serving layer stateless so it can scale horizontally — and place conversation state where it actually belongs
- Make an agent service survive load: upstream rate limits, concurrency control, backpressure, timeouts, and retries with backoff
- Package an agent as a container, configure it the 12-factor way, add health checks and graceful shutdown, and roll it out without downtime
</LearningObjectives>

## What you'll build

A production-ready service wrapping the Module 3 research agent: a containerized FastAPI app exposing a synchronous endpoint and a streaming (SSE) endpoint, with stateless request handling, health/readiness probes, graceful shutdown, env-based config, and retries around the model call. The Module 4 eval suite gates the build — the image only ships if the agent still passes its evals.

## Why deployment is its own discipline

The agent that works in your notebook works because *you* are the runtime: you hold the API key, you wait for it to finish, you re-run the cell when it errors, and only one request exists at a time. Production removes all four of those crutches at once. Many requests arrive concurrently, the network fails, the model rate-limits you, the process gets killed mid-request during a deploy, and the secret can't live in the source. This module is about the engineering that stands between "it runs on my machine" and "it serves thousands of users without you watching."
