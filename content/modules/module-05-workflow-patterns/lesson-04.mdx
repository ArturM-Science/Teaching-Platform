# Lesson 4 - Loops, Repair, and Stopping Conditions

A single pass through a workflow is not always enough. Sometimes the output needs to be checked and corrected before it is acceptable. Loops let a workflow re-run a step — or a sequence of steps — until a condition is met. The risk is that without a stopping condition, a loop runs forever.

## The verification loop

The core pattern: generate, verify, repair, repeat.

```
[generator] → output
                ↓
          [verifier] → pass? → done
                ↓ fail
          [repair prompt] → revised output
                ↓
          [verifier] → pass? → done
                ↓ fail (iteration 2)
          [repair prompt] → ...
```

The verifier is separate from the generator. It has one job: check a specific criterion and return a pass/fail signal. It should not also generate a fix — that is the repair step's job. Mixing verification and repair into one step produces outputs that are harder to test independently.

## Self-correction vs external validation

Two places to run verification:

**Self-correction**: Ask the same model that produced the output to review and revise it.

```text
Review your previous answer. Does it include a citation for every factual claim?
If not, add the missing citations. Return the revised answer only.
```

**External validation**: Use a separate call, tool, or deterministic check.

```python
# Deterministic: count citations
if citation_count(output) < claim_count(output):
    trigger_repair()

# Separate model call: independent review
verifier_result = verifier_model(output, criteria)
```

External validation is stronger than self-correction. A model that hallucinated a fact is likely to confirm it when asked to verify. A separate check — especially a deterministic one — is not subject to the same bias.

| Approach | Strength | Weakness |
|---|---|---|
| Self-correction | Simple, no extra call | Model confirms its own errors |
| External model | Independent perspective | Higher latency and cost |
| Deterministic check | Reliable, cheap | Only works for measurable criteria |

Use deterministic checks when the criterion is checkable in code. Use an external model when the criterion is qualitative. Combine both for high-stakes outputs.

## Stopping conditions — the mandatory gate

Every loop must have at least one stopping condition that does not depend on the model deciding it is done.

Three acceptable stopping conditions:

1. **Max iterations**: `if iteration >= MAX_ITERATIONS: break`
2. **Budget**: `if total_tokens >= TOKEN_BUDGET: break`
3. **Explicit success signal**: the verifier returns `{"pass": true}`

Without a stopping condition, a loop that encounters a persistent failure — a hallucination the model cannot self-correct, an edge case the verifier always rejects — will run until an external limit (rate limit, timeout, cost cap) terminates it. That external limit is painful and unpredictable.

<Callout type="key">
Every loop needs an exit condition that does not depend on the model saying it is done. The model's "I'm satisfied" is not a reliable stopping signal.
</Callout>

## Setting the iteration limit

Set the limit based on the task, not on a round number:

- **One repair pass** is right for most cases. If the output does not pass after one revision, the generator or verifier prompt is probably wrong.
- **Two to three passes** are appropriate when the criteria are complex and incremental improvement is expected.
- **More than three passes** usually signals that the generator and verifier are in a loop that cannot converge. Stop, log, and route to a fallback or human review.

```python
MAX_ITERATIONS = 2
iteration = 0

output = generator(input)
while iteration < MAX_ITERATIONS:
    result = verifier(output)
    if result["pass"]:
        break
    output = repair(output, result["feedback"])
    iteration += 1

if not result["pass"]:
    escalate_to_human(input, output, result)
```
