# Lesson 3 - Branching, Routing, and Classifiers

Not every input should follow the same path. A support ticket about a billing error needs a different handler than one about a login failure. Branching adds conditional logic to a workflow — the path taken depends on what was observed at a prior step.

## What branching adds

A linear chain runs every step for every input. Branching routes inputs to different steps based on a condition:

```
input → [classifier] → category
                          ↓
              billing → [billing_handler]
              technical → [technical_handler]
              general → [general_handler]
```

The classifier is cheap to run and produces a structured decision. The handlers are more expensive and specialised. This pattern pays off when:

- Different inputs genuinely need different processing
- One handler would produce worse output than a specialist for a given category
- You want to apply different safety checks, templates, or tools depending on input type

## The classifier → specialist pattern

The most common routing pattern in agent workflows:

1. **Classifier step**: a fast, low-cost call that reads the input and returns a category label (and optionally a confidence score)
2. **Specialist step**: a separate, purpose-built prompt or agent that handles one category well

```text
Classify the following support message into exactly one of: billing, technical, general.
Return JSON: {"category": "...", "confidence": 0.0}

Message: {input}
```

The classifier output is machine-readable, which makes the routing decision deterministic in code — not left to the next model to interpret.

```python
result = classifier(input)
if result["category"] == "billing":
    return billing_handler(input)
elif result["category"] == "technical":
    return technical_handler(input)
else:
    return general_handler(input)
```

## Fallback chains

Not every path succeeds on the first attempt. A fallback chain tries the primary path; if it fails or returns low confidence, it falls back to a safer alternative.

```
input → [primary handler]
              ↓ (if confidence < 0.7 or error)
        [fallback handler]
              ↓ (if still unresolved)
        [escalate to human]
```

Fallback chains are especially useful when:

- The primary handler uses an expensive model that occasionally fails
- Low-confidence outputs should not be sent to the user without review
- Certain error types need human eyes regardless of model confidence

## Making routing decisions auditable

Every branch decision should be logged. When a bug report arrives saying "the wrong template was sent to a customer," you need to know what the classifier returned, not just what the handler produced.

Minimum logging at each branch point:

```json
{
  "input_hash": "a3f9...",
  "classifier_output": {"category": "billing", "confidence": 0.91},
  "branch_taken": "billing_handler",
  "timestamp": "2026-05-31T09:14:00Z"
}
```

This gives you a trace through the workflow for every input — essential for debugging and for catching silent misrouting.
