# Lesson 2 - Sequential Chains and Context Passing

The simplest workflow is a chain: step one runs, its output becomes the input to step two, and so on. The engineering challenge is not wiring the steps together — it is deciding what to carry forward and what to drop.

## The basic chain

```
question → [researcher] → raw_notes → [writer] → draft → [editor] → final
```

Each step receives input and produces output. The output is passed to the next step as part of its prompt. Three things to decide at each handoff:

1. **What does the next step need?** Not everything the previous step produced.
2. **What format should it be in?** Structured data (JSON, bullet list) is easier to consume than free prose.
3. **How much of the earlier context should travel forward?** Less is usually better.

## The context accumulation problem

The naive approach is to append every step's output into a growing context string and pass the whole thing forward. This breaks in two ways:

**Window overflow**: After several steps, the accumulated context exceeds the model's context window. The model either errors, truncates silently, or degrades on the content it keeps.

**Quality degradation**: Long contexts with irrelevant earlier output dilute the signal for later steps. The model attends to the wrong things.

```
step 1 output: 2,000 tokens
step 2 output: 2,000 tokens
step 3 input: 4,000 tokens of prior output + 500 token prompt = 4,500 tokens
step 4 input: 6,500 tokens of prior output + 500 token prompt = 7,000 tokens
...
```

At ten steps, a chain with no compression consumes 20,000+ tokens of context that is mostly noise.

<Callout type="warning">
Carrying raw step output forward across many steps bloats context and degrades quality. Every handoff is a design decision about what the next step actually needs.
</Callout>

## The compression step pattern

Insert a compression step between stages to reduce context before the next step consumes it.

```
[researcher] → raw_notes (2,000 tokens)
             → [compressor] → summary (300 tokens)
             → [writer] → draft
```

The compressor's job is explicit: reduce the prior output to what the next step needs, in the format the next step expects. A good compressor prompt:

```text
Summarise the following research notes for a writer who needs to draft a 500-word explainer.

Include:
- The three most important facts with their sources
- Any caveats or contradictions
- Nothing else

Notes:
{raw_notes}
```

## Structured handoffs

Prose is hard for downstream steps to parse reliably. Prefer structured output at each handoff:

```json
{
  "category": "billing",
  "confidence": 0.94,
  "key_facts": ["account overcharged $47", "charge date: 2026-05-01"],
  "recommended_template": "billing_overcharge"
}
```

A downstream step consuming this JSON can extract exactly what it needs without parsing prose. It also makes the handoff testable: you can run the upstream step, inspect the JSON, and verify it before wiring in the downstream step.

## Prompt chaining vs single long prompt

| Approach | When to prefer |
|---|---|
| Single long prompt | Task is self-contained; one model can hold the full context |
| Prompt chain | Steps have different goals; output of one is input to another; verification between steps adds value |

Prompt chaining is not always better. A chain that calls the API four times for a task a single prompt handles has four times the latency and four times the failure surface. Use chains when the step separation is real, not cosmetic.
