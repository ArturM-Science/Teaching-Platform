## What you will build

A research agent that:
1. Accepts a research query and searches 3 web sources
2. Streams its reasoning and tool calls to a React frontend in real time
3. Pauses before writing a final report to a file (irreversible) and waits for user approval with full consequence context
4. Accepts a thumbs-down correction after the report is written and reruns the report with the corrected context
5. Degrades gracefully if a source is unavailable — shows partial results with a clear note, does not abort

## Part 1 — Streaming backend (FastAPI + SSE)

Create a FastAPI endpoint that runs the agent and emits SSE events.

### Endpoint

```
POST /api/research
Content-Type: application/json
Body: { "query": string }

Response: text/event-stream
```

### Event format

Each event is a JSON object on a `data:` line, followed by `\n\n`:

```
data: {"type": "token", "content": "The first result shows..."}

data: {"type": "tool_call", "tool": "web_search", "input": {"query": "climate tipping points 2024"}}

data: {"type": "tool_result", "tool": "web_search", "summary": "Found 3 results", "count": 3}

data: {"type": "approval_required", "action": "write_report", "filename": "report.md", "preview": "# Research Report\n...", "consequence": "This will write a file to your workspace. This action can be undone by deleting the file."}

data: {"type": "done", "steps_completed": 5, "elapsed_ms": 8400}

data: {"type": "error", "message": "Source 2 timed out", "recoverable": true, "suggestion": "The partial report excludes source 2. You can retry with source 2 only."}
```

### Agent loop pseudocode

```python
async def research_agent(query: str, stream: AsyncGenerator):
    await stream({"type": "token", "content": "Planning research..."})

    sources = await search_web(query, count=3)          # tool_call + tool_result events
    summaries = []

    for i, source in enumerate(sources):
        try:
            summary = await read_source(source)         # tool_call + tool_result events
            summaries.append(summary)
        except TimeoutError:
            await stream({"type": "error", "message": f"Source {i+1} timed out", "recoverable": True, ...})
            # continue loop — graceful degradation

    # Stream the report draft token by token
    report = await compose_report(query, summaries, stream=stream)

    # Pause for approval before writing to disk
    await stream({
        "type": "approval_required",
        "action": "write_report",
        "filename": f"{slugify(query)}.md",
        "preview": report[:500],
        "consequence": "This will write a Markdown file to your workspace."
    })

    # Wait for approval signal (use a queue or SSE + POST pattern)
    approved = await wait_for_approval()

    if approved:
        write_file(report)
        await stream({"type": "done", ...})
    else:
        await stream({"type": "token", "content": "Report cancelled. The draft is shown above."})
        await stream({"type": "done", ...})
```

**Approval gate pattern:** The frontend sends a `POST /api/research/{run_id}/approve` or `/cancel` after the user responds to the approval dialog. The backend holds the run in a waiting state (use `asyncio.Event` or a Redis queue).

## Part 2 — React streaming frontend

### Component structure

```
<ResearchPage>
  ├── <QueryInput />               query form
  ├── <StatusBar status={...} />   current agent state
  ├── <ToolCallFeed events={...} /> live tool call cards
  ├── <TokenStream content={...} /> streaming answer text
  ├── <ApprovalDialog />           shown when approval_required event arrives
  └── <FeedbackBar />              thumbs up/down + correction input
```

### SSE client

```typescript
async function runResearch(query: string, handlers: StreamHandlers) {
  const res = await fetch('/api/research', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ query }),
  })

  const reader = res.body!.getReader()
  const decoder = new TextDecoder()

  while (true) {
    const { done, value } = await reader.read()
    if (done) break

    const lines = decoder.decode(value).split('\n\n')
    for (const line of lines) {
      if (!line.startsWith('data:')) continue
      const event = JSON.parse(line.slice(5).trim())
      handlers[event.type]?.(event)
    }
  }
}
```

### Approval gate UX

When the `approval_required` event arrives:
- Render an `<ApprovalDialog>` overlay (not a modal that blocks the tool feed — overlay so the context is visible)
- Show: action name, filename, full consequence text, a collapsible report preview (first 500 chars + "Show full report" expander)
- Two buttons: "Write report" (primary) and "Cancel" (secondary)
- No third button. No "Edit" for this lab.

After the user responds, POST to `/api/research/{run_id}/approve` or `/cancel` and hide the dialog.

## Part 3 — Feedback and correction

After the run completes, show the feedback bar:

```
[👍] [👎]  Was this report useful?
```

On thumbs-down:
1. Show an inline textarea: "What was wrong? (optional)"
2. Submit button
3. On submit: POST to `/api/research/{run_id}/feedback` with `{ "rating": -1, "correction": "..." }`
4. If a correction was provided, offer a "Rerun with correction" button
5. On "Rerun with correction": send a new research request with the original query + correction appended to context

Rerunning with a correction is a new run, not a continuation. For this lab, appending "User correction: {text}" to the query context is sufficient.

## Part 4 — Graceful degradation

Your agent must not abort when a single source is unavailable. Test this by mocking a `TimeoutError` on source 2 in your source reader.

Expected behaviour:
- The tool call card for source 2 shows "Timed out — excluded from report"
- The report is generated from sources 1 and 3
- A banner above the report reads: "Note: source 2 was unavailable. This report covers 2 of 3 sources."
- The approval gate still fires before writing
- The final file includes a front-matter note: `sources_available: 2/3`

<Solution>

#### The approval-wait pattern (the part everyone gets stuck on)

The SSE stream and the approve button arrive on *different HTTP requests*, so they need a shared rendezvous keyed by run id:

```python
# state shared between the stream and the approve/cancel endpoints
import asyncio, uuid
from fastapi import FastAPI, HTTPException

app = FastAPI()
RUNS: dict[str, dict] = {}   # run_id -> {"event": asyncio.Event, "approved": bool}

async def wait_for_approval(run_id: str, timeout: float = 300) -> bool:
    run = RUNS[run_id]
    try:
        await asyncio.wait_for(run["event"].wait(), timeout=timeout)
    except asyncio.TimeoutError:
        return False                      # timeout = cancel, never auto-approve
    return run["approved"]

@app.post("/api/research/{run_id}/approve")
async def approve(run_id: str):
    run = RUNS.get(run_id)
    if not run:
        raise HTTPException(404)
    run["approved"] = True
    run["event"].set()
    return {"ok": True}

@app.post("/api/research/{run_id}/cancel")
async def cancel(run_id: str):
    run = RUNS.get(run_id)
    if not run:
        raise HTTPException(404)
    run["approved"] = False
    run["event"].set()
    return {"ok": True}
```

In the `/api/research` handler, create `run_id = uuid.uuid4().hex`, register `RUNS[run_id] = {"event": asyncio.Event(), "approved": False}`, and **include `run_id` in the `approval_required` event** so the frontend knows where to POST — that's the missing field in the starter's event format. Clean up `RUNS.pop(run_id, None)` in a `finally`. Two design points worth writing down: a timeout resolves to *cancel* (irreversible actions fail closed), and this dict only works single-process — multiple workers need the Redis variant.

The report prompt then receives only `summaries`, and the file write prepends front-matter: `sources_available: {len(summaries)}/{len(sources)}`. The principle being tested: degradation is *declared*, in the stream, in the banner, and in the artifact — partial results that look complete are worse than failures.

#### Feedback rerun

```python
@app.post("/api/research/{run_id}/feedback")
async def feedback(run_id: str, body: dict):
    log_feedback(run_id, body["rating"], body.get("correction", ""))
    return {"ok": True}
```

The rerun button simply calls `runResearch` again with:

```typescript
const rerunQuery = `${originalQuery}\n\nUser correction from previous attempt: ${correction}`
```

New run, new run_id, full pipeline including the approval gate — a correction does not inherit the previous run's approval, because the report it produces is different.

#### Frontend: the three state transitions that matter

```typescript
const handlers: StreamHandlers = {
  token:            e => setAnswer(a => a + e.content),
  tool_call:        e => setFeed(f => [...f, { ...e, status: "running" }]),
  tool_result:      e => setFeed(f => markDone(f, e.tool)),
  error:            e => setFeed(f => [...f, { type: "error", label: e.message, note: e.suggestion }]),
  approval_required: e => setApproval(e),       // renders <ApprovalDialog {...e} />
  done:             e => { setApproval(null); setShowFeedback(true) },
}
```

The `error` handler appending a labelled card (instead of throwing or toasting a stack trace) is what satisfies the checkpoint's "no stack traces in the UI under any failure scenario" — every failure path ends in a rendered, human-readable element.
