# Lesson 3 · Human-in-the-Loop Patterns

Giving humans control over agents is not about distrust. It is about matching the level of autonomy to the level of risk. An approval gate on a routine query is friction. An approval gate before an irreversible action is safety.

<LearningObjectives>
- Name and distinguish the four HITL patterns: approval gate, interrupt, escalation, and handoff
- Choose the correct pattern for a given risk level and task type
- Design an approval gate that avoids the rubber-stamp failure mode
- Explain why interrupt points belong at natural task boundaries
</LearningObjectives>

## The four HITL patterns

**Approval gate** — the agent pauses before executing a high-risk or irreversible action and waits for explicit user confirmation. The user can approve, reject, or modify the proposed action. The agent resumes or aborts based on the response.

**Interrupt** — the user pauses a running agent mid-execution. The agent must be designed with well-defined interrupt points — places where it can safely stop without leaving shared state inconsistent. After an interrupt, the user can redirect or resume.

**Escalation** — the agent-initiated version of interrupt. When the agent encounters a situation outside its confidence range (ambiguous instruction, edge case, conflicting constraints), it pauses and transfers control to a human with a summary of what it knows so far.

**Handoff** — the task is transferred to a human permanently. The agent composes a full context packet — what was done, what remains, what decisions need to be made — and passes it to a human queue or inbox.

<Callout type="key">
Design the interrupt point first, then the approval copy. If you cannot write the consequence in one sentence, your action scope is too large.
</Callout>

## The rubber-stamp problem

The most common failure mode in approval gate design is the **rubber-stamp**: an approval dialog with insufficient context that users approve automatically without reading.

A poorly designed gate shows: "Send email?" with Approve / Reject buttons.

A well-designed gate shows:
- **What**: "Send a marketing email"
- **To whom**: "5,823 subscribers in the UK segment"
- **Content preview**: subject line and first 100 characters of body
- **Consequence**: "This action cannot be undone. Replies will go to your connected inbox."
- **Alternatives**: "Edit email · Change segment · Schedule instead"

The rule: if you cannot write the consequence in one sentence, the action scope is too large. Break it into smaller actions, each with a narrower gate.

## Interrupt design

Interrupts must be planned at design time, not added as an afterthought. An agent that can be interrupted anywhere is an agent that may be interrupted in the middle of a multi-step database write, leaving state inconsistent.

Well-designed interrupt points:
- Fall at natural task boundaries: after a tool call completes, before the next one starts
- Leave the agent state serialisable: everything the agent knows can be saved to a store
- Are visible to the user: "Pause after this step" is a UI control, not a hidden capability

Poorly-designed interrupt points:
- Mid-tool-call (dangerous: partial side effects)
- After a deadline-sensitive external API call (dangerous: resources held)
- At no defined points at all (the interrupt button aborts the run instead of pausing it)

**Escalation** is the agent's version of "I don't know how to proceed." It should include: what the agent has done, what the ambiguity is (in plain language), and what the human needs to decide. It is not an error — it is the agent correctly recognising the limits of its autonomy.
