# Lab - Configure a Safe Claude Code Workflow

<LabBrief title="Safe Claude Code Workflow" time="2-3 hours">

#### Objective

Configure Claude Code for one real repository, complete a small verified change, and design how the same workflow would look as a Claude Managed Agents deployment.

#### Setup

Choose a repository you are allowed to edit. Start from a clean branch:

```bash
git status --short
git switch -c claude-code-safe-workflow
```

Use Claude Code in `plan` or `default` mode first. Do not start this lab in `bypassPermissions`.

#### Deliverables

- A small final diff that can be reviewed in under 10 minutes
- A `CLAUDE.md` or `.claude/rules/` update, or a note explaining why none was appropriate
- One permission rule or hook configuration proposal
- Verification output from a real command
- A Claude Code review pass
- `claude-code-lab-notes.md` with prompts, decisions, and Managed Agents design notes

#### Hard constraints

- Do not read secrets, private keys, or production data
- Do not add dependencies without stopping for human approval
- Do not use `bypassPermissions`
- Do not run deployment, migration, or external write commands
- Final diff must be small and reviewable

</LabBrief>

## Step 1: Explore in plan mode

Ask Claude Code to identify three candidate tasks:

```text
/plan

Inspect this repository and propose three small improvements.

Each candidate must:
- be reviewable in under 10 minutes
- have a clear verification command
- avoid auth, billing, production data, secrets, and deployment
- require no new dependency

Do not edit files.

Return:
1. Task
2. Files likely involved
3. Verification command
4. Risk level
5. Why this is small enough
```

Pick one. If every task is too broad, ask for smaller candidates.

## Step 2: Create lab notes

Create `claude-code-lab-notes.md`:

```md
# Claude Code lab notes

## Chosen task

## Files in scope

## Non-goals

## Permission mode

## Verification command

## Prompts used

## Review findings

## CLAUDE.md / rules / hook decision

## Managed Agents design

## Reflection
```

## Step 3: Add durable context

Ask Claude Code whether the repo has a useful `CLAUDE.md`:

```text
Inspect existing project guidance.

Tell me:
1. Whether CLAUDE.md exists.
2. Whether AGENTS.md exists and should be imported.
3. One specific instruction that would help future Claude Code sessions.

Do not edit until I approve.
```

Good updates:

```md
@AGENTS.md

## Claude Code

- Use plan mode before touching `app/auth/` or Supabase policies.
- Run `npm run build` after MDX rendering changes.
- Do not add dependencies without asking first.
```

If no durable rule emerged, record that in the notes.

## Step 4: Implement the small change

Give Claude Code a bounded implementation prompt:

```text
Implement the chosen task.

Scope:
- [files or directory]

Non-goals:
- Do not refactor unrelated code.
- Do not change auth, billing, database policies, deployment, or secrets.
- Do not add dependencies.

Verification:
- Run [command].
- If it fails, diagnose root cause before editing again.
- Rerun the same command after the fix.

Report:
- Files changed
- Verification result
- Remaining test gap
```

## Step 5: Add a permission rule or hook proposal

You do not need to install a hook during the lab, but you must propose one useful rule or hook.

Examples:

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Read(./.env)",
      "Bash(git push *)"
    ]
  }
}
```

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint -- --fix"
          }
        ]
      }
    ]
  }
}
```

If a hook would be too risky or slow, explain why a permission rule is better.

## Step 6: Review the diff

Ask for a blocking review:

```text
Review the current diff as if blocking a pull request.

Findings first, ordered by severity.
Prioritize bugs, security, data exposure, missing tests, and wrong framework assumptions.
If there are no blocking findings, say that clearly and name residual risk.
Do not edit files during review.
```

If Claude finds a real issue:

```text
Fix only finding #1. Do not change unrelated files. Rerun the verification command.
```

The goal is not to deploy Managed Agents in this lab. The goal is to prove you understand the architectural translation.

Submit:

- Final diff
- `claude-code-lab-notes.md`

- review output
- `CLAUDE.md` or rule change, if any

- Managed Agents design sketch
