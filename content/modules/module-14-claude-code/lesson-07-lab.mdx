# Lab - Configure a Safe Claude Code Workflow

<LabBrief title="Safe Claude Code Workflow" time="2-3 hours">

#### Objective

Configure Claude Code for one real repository, complete a small verified change, and design how the same workflow would look as a Claude Managed Agents deployment.

#### Setup

Choose a repository you are allowed to edit. Start from a clean branch:

```bash
git status --short
git switch -c claude-code-safe-workflow
```

Use Claude Code in `plan` or `default` mode first. Do not start this lab in `bypassPermissions`.

#### Deliverables

- A small final diff that can be reviewed in under 10 minutes
- A `CLAUDE.md` or `.claude/rules/` update, or a note explaining why none was appropriate
- One permission rule or hook configuration proposal
- Verification output from a real command
- A Claude Code review pass
- `claude-code-lab-notes.md` with prompts, decisions, and Managed Agents design notes

#### Hard constraints

- Do not read secrets, private keys, or production data
- Do not add dependencies without stopping for human approval
- Do not use `bypassPermissions`
- Do not run deployment, migration, or external write commands
- Final diff must be small and reviewable

</LabBrief>

## Step 1: Explore in plan mode

Ask Claude Code to identify three candidate tasks:

```text
/plan

Inspect this repository and propose three small improvements.

Each candidate must:
- be reviewable in under 10 minutes
- have a clear verification command
- avoid auth, billing, production data, secrets, and deployment
- require no new dependency

Do not edit files.

Return:
1. Task
2. Files likely involved
3. Verification command
4. Risk level
5. Why this is small enough
```

Pick one. If every task is too broad, ask for smaller candidates.

## Step 2: Create lab notes

Create `claude-code-lab-notes.md`:

```md
# Claude Code lab notes

## Chosen task

## Files in scope

## Non-goals

## Permission mode

## Verification command

## Prompts used

## Review findings

## CLAUDE.md / rules / hook decision

## Managed Agents design

## Reflection
```

## Step 3: Add durable context

Ask Claude Code whether the repo has a useful `CLAUDE.md`:

```text
Inspect existing project guidance.

Tell me:
1. Whether CLAUDE.md exists.
2. Whether AGENTS.md exists and should be imported.
3. One specific instruction that would help future Claude Code sessions.

Do not edit until I approve.
```

Good updates:

```md
@AGENTS.md

## Claude Code

- Use plan mode before touching `app/auth/` or Supabase policies.
- Run `npm run build` after MDX rendering changes.
- Do not add dependencies without asking first.
```

If no durable rule emerged, record that in the notes.

## Step 4: Implement the small change

Give Claude Code a bounded implementation prompt:

```text
Implement the chosen task.

Scope:
- [files or directory]

Non-goals:
- Do not refactor unrelated code.
- Do not change auth, billing, database policies, deployment, or secrets.
- Do not add dependencies.

Verification:
- Run [command].
- If it fails, diagnose root cause before editing again.
- Rerun the same command after the fix.

Report:
- Files changed
- Verification result
- Remaining test gap
```

## Step 5: Add a permission rule or hook proposal

You do not need to install a hook during the lab, but you must propose one useful rule or hook.

Examples:

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Read(./.env)",
      "Bash(git push *)"
    ]
  }
}
```

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint -- --fix"
          }
        ]
      }
    ]
  }
}
```

If a hook would be too risky or slow, explain why a permission rule is better.

## Step 6: Review the diff

Ask for a blocking review:

```text
Review the current diff as if blocking a pull request.

Findings first, ordered by severity.
Prioritize bugs, security, data exposure, missing tests, and wrong framework assumptions.
If there are no blocking findings, say that clearly and name residual risk.
Do not edit files during review.
```

If Claude finds a real issue:

```text
Fix only finding #1. Do not change unrelated files. Rerun the verification command.
```

<Solution label="Show a worked example session">

Each run differs — this shows the two deliverables learners most often under-deliver: the safety-control proposal (Step 5) and the Managed Agents design (Step 7), filled in for a real session on a Next.js content site.

#### Step 5 — a permission proposal tied to an actual risk

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run build)",
      "Bash(npm run lint *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(git log *)"
    ],
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Bash(git push *)",
      "Bash(npx supabase *)"
    ]
  }
}
```

With the justification from the notes: *"The allow list is exactly the verification loop this repo uses, so routine sessions need zero prompts. The deny list maps to the two real risks observed: the repo's `.env.local` holds a Supabase service key (data exposure), and `npx supabase db push` is an external write that no local task should ever need. I chose permission rules over a PostToolUse lint hook because lint here takes 40s — as a hook it would fire on every edit and make Claude slower without blocking anything dangerous."*

That last sentence is what "Exceeds" wants: the *rejected* option is named with a reason. A proposal with no rejected alternative reads as a template paste.

#### Step 7 — Managed Agents design at scoring depth

```md
## Managed Agents design

### Agent
- Model: claude-sonnet-4-6 (content edits don't need Opus; cost matters at cron frequency)
- System prompt: the repo's CLAUDE.md rules + "produce a PR, never push to main"
- Skills: none required
- MCP servers: none — repo file access only
- Built-in tools enabled: file read/write/edit, bash (build + lint only)
- Custom tools: none

### Environment
- Cloud sandbox (Anthropic-hosted): the repo is on GitHub, no internal services needed
- Network policy: GitHub + npm registry only; all other egress blocked
- Secrets policy: a fine-grained GitHub token scoped to this repo, contents+PR
  only; no Supabase credentials enter the environment (builds use a stub env)

### Session
- User event: weekly cron (Mon 07:00) + manual trigger from the dashboard
- Success criteria: lint and build pass, diff < 200 lines, PR opened with summary
- Stop conditions: 30 min wall clock, $2 budget, or any permission denial
- Human approval points: PR review is the single gate — the agent has no merge rights

### Permission policy
- always_allow: read repo, edit content/**, run npm run build / lint
- always_ask: edits outside content/**, any new dependency, any git push

### Data handling
- Stored server-side: session transcripts and diffs (30-day retention)
- Must not enter the session: .env values, Supabase keys, user PII from the prod DB
- Deletion/retention: transcripts auto-expire; PRs are the durable audit record
```

The translation insight that separates a 4 from a 3, stated in one line in the notes: *"Locally, I am the approval gate at every permission prompt; in Managed Agents, nobody is watching mid-session — so every mid-session 'ask me' either becomes a hard deny, a pre-approved allow, or a stop condition. The PR boundary replaces me."* If your design still contains interactive approval points with no one to answer them, the translation isn't done.

#### Common pitfalls in the rest of the lab

- **Step 3:** the most frequent durable-context mistake is writing a CLAUDE.md rule about *this session's task* ("the port flag is in cli.py"). The repo records that already — durable rules capture what the repo *can't* say about itself (build quirks, forbidden zones, verification habits).
- **Step 6:** if the review pass returns "looks good" in one line, it didn't engage — rerun with the blocking phrasing. An explicit "no blocking findings; residual risk: no test covers the empty-input path" is a *pass*; silence is not.

</Solution>
