# Lesson 2 - Prompting, Plan Mode, Checkpoints, and Verification

Claude Code is conversational, but good sessions still need engineering discipline. The pattern is simple: explore first when uncertain, implement with a boundary, verify with a command, and review the diff.

## Prompt shape

A strong Claude Code prompt contains five parts:

1. **Goal** - what should be true after the change.
2. **Context** - files, errors, screenshots, logs, relevant docs.
3. **Boundary** - what is in scope and out of scope.
4. **Verification** - command, test, build, or manual check.
5. **Reporting** - what you want back at the end.

```text
Goal:
Fix the failing lesson render when an MDX component receives an array prop.

Context:
- Rendering uses next-mdx-remote/rsc.
- Read AGENTS.md before editing.
- The symptom is Quiz options becoming undefined.

Boundary:
- Start in app/(learner)/modules/[module]/[lesson]/page.tsx.
- Do not change MDX content or Quiz.tsx unless the root cause requires it.

Verification:
- Run npm run build.
- If build passes, compile one affected MDX lesson directly if possible.

Report:
- Root cause
- Files changed
- Verification result
- Remaining test gap
```

<Callout type="tip">
If you are not sure where the bug lives, do not ask for a fix first. Ask Claude Code to investigate and propose the smallest plan.
</Callout>

## Plan mode

Plan mode lets Claude read and investigate before editing. It is the right default for:

- unfamiliar repositories
- security-sensitive areas
- large migrations
- ambiguous product requests
- bugs without a known reproduction

```text
/plan

Investigate why the dashboard shows modules with content as locked.

Return:
1. The files involved
2. The current control flow
3. The smallest likely fix
4. The verification command

Do not edit files until I approve the plan.
```

Plan mode is not bureaucracy. It is how you separate understanding from mutation.

## Interrupt and steer

Claude Code does not need to finish a wrong path. You can interrupt and redirect:

```text
Stop. This is broader than the task.

Only fix the failing test in test_content_loader.py.
Do not refactor the loader.
Rerun the same test after the fix.
```

Good steering is specific. "No, not that" is less useful than "do not touch auth; the issue is the route guard."

## Checkpoints and Git

Claude Code creates local checkpoints before file edits, but Git is still your review surface.

Before a real task:

```bash
git status --short
git switch -c claude-code/mdx-array-props
```

During a task:

```bash
git diff --stat
git diff
```

After a task:

```bash
npm run build
git diff --check
```

Use checkpoints to rewind a bad agent turn. Use Git to review, commit, branch, and merge.

## Verification loops

Claude Code performs best when the verification signal is objective:

| Task | Better verification |
|---|---|
| Bug fix | Re-run the failing test or reproduction |
| TypeScript change | `npm run build` or type check |
| Formatting change | formatter plus `git diff --check` |
| UI behavior | screenshot or Playwright check |
| Docs change | link check or MDX compile |

When verification fails, use a repair prompt:

```text
The verification command failed.

Command:
npm run build

Relevant output:
[paste output]

Diagnose the root cause before editing. Make the smallest fix, then rerun the same command. If the same error appears twice, stop and explain the blocker.
```
