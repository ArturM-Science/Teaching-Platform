# Lesson 1 · Context Is Not Memory

An agent that answered your questions perfectly in session 1 has no idea what you discussed in session 2. You didn't change the model. You didn't change the prompt. The context window was just cleared. That's not a bug — it's the architecture. Understanding why is the starting point for building agents that actually remember things.

## The context window is a whiteboard

Every LLM call is stateless. When you send a request, you pass the entire conversation history in the `messages` array. The model reads it, generates a response, and discards everything. Nothing persists. The "memory" you see in a chat interface is an illusion maintained by the application layer — it appends each turn to the messages list and resends the whole thing every time.

This creates two hard constraints:

**Capacity.** Context windows are large but finite. GPT-4o has 128k tokens; Claude 3 Opus has 200k. A single book is roughly 100k tokens. Long-running agents accumulate conversation history, tool outputs, and reasoning traces that eventually exceed any context limit.

**Cost.** Every token in the context window is processed on every call. A 100k-token context costs roughly 100× more per call than a 1k-token context. Naively appending everything to context is expensive even before you hit the limit.

```python
# What "memory" looks like without any memory system
messages = [
    {"role": "system", "content": "You are a helpful assistant."},
    # Every prior turn is re-sent on every call
    {"role": "user",    "content": "My name is Artur."},
    {"role": "assistant","content": "Nice to meet you, Artur."},
    {"role": "user",    "content": "What's my name?"},
]
# This works — until the conversation exceeds the context limit
# Then you either truncate (losing information) or fail
```

<Callout type="key">
The context window is not memory. It is a read-once scratchpad that is rebuilt from scratch on every single API call. Treating it as memory works until it doesn't — and when it fails, it fails silently by truncating the history you were relying on.
</Callout>

## Context management strategies

Before reaching for a memory system, there are three lighter techniques for managing context pressure:

**Sliding window.** Keep only the last N turns. Simple, but loses older context entirely. Works for tasks where recent history is sufficient.

```python
MAX_TURNS = 20

def trim_messages(messages: list[dict], system: str) -> list[dict]:
    non_system = [m for m in messages if m["role"] != "system"]
    trimmed = non_system[-MAX_TURNS:]
    return [{"role": "system", "content": system}] + trimmed
```

**Summarisation.** When the conversation exceeds a threshold, use an LLM call to compress older turns into a summary, then replace them with it. Retains semantic content at the cost of an extra API call.

```python
def summarise_and_trim(messages: list[dict], threshold: int = 8000) -> list[dict]:
    token_count = sum(len(m["content"].split()) for m in messages)
    if token_count < threshold:
        return messages

    older = messages[:-10]  # keep the last 10 turns verbatim
    recent = messages[-10:]

    summary_text = call_llm(
        f"Summarise this conversation history concisely:\n\n"
        + "\n".join(f"{m['role']}: {m['content']}" for m in older)
    )
    summary_msg = {"role": "user", "content": f"[Earlier conversation summary: {summary_text}]"}
    return [summary_msg] + recent
```

**Retrieval.** Don't keep history in context at all — store it externally and retrieve only the relevant parts per turn. This is what a proper memory system does, and it's the subject of the rest of this module.

## Memory taxonomy

Once you accept that context isn't memory, you need to decide what kind of memory to build. There are three distinct types, and choosing the wrong one for a given piece of information is a common design mistake.

**Semantic memory** stores facts about the world — things that are true independent of any particular interaction. Product documentation, user preferences, domain knowledge. This is what a vector database is for.

```python
class SemanticMemory:
    """Facts that are true regardless of when or who asked."""
    def store(self, fact: str, metadata: dict): ...
    def search(self, query: str, k: int = 5) -> list[str]: ...
```

**Episodic memory** stores past interactions — what was discussed, what happened, what was decided. The history of sessions with a specific user. This is what session logs and conversation summaries capture.

```python
class EpisodicMemory:
    """What happened in past sessions with this user."""
    def log_session(self, session_id: str, summary: str, date: str): ...
    def recall(self, user_id: str, n_sessions: int = 3) -> list[str]: ...
```

**Procedural memory** stores how to do things — patterns, templates, successful reasoning traces that should be reused. Few-shot examples are a form of procedural memory.

```python
class ProceduralMemory:
    """How to handle known task types."""
    def store_pattern(self, task_type: str, successful_trace: str): ...
    def retrieve_pattern(self, task_type: str) -> str | None: ...
```

<Callout type="warning">
These types blur in practice. A user's name could be semantic (a fact about the world) or episodic (learned in session 1). The distinction clarifies when you hit an actual failure: "the agent forgot we discussed this" is episodic failure; "the agent doesn't know anything about our product" is semantic failure. Diagnose which type is missing before building the wrong system.
</Callout>

## Which type to reach for

The choice depends on what the agent is forgetting:

| What's missing | Memory type | Storage |
|---|---|---|
| Facts, domain knowledge, documentation | Semantic | Vector DB (ChromaDB) |
| Past conversations, user history, session context | Episodic | Database, vault, logs |
| Reasoning patterns, successful task traces | Procedural | Few-shot examples, prompt library |
| All three across sessions | Hybrid | Vault (file system) + vector index |

The lab at the end of this module builds a hybrid system: an Obsidian vault as the durable store and ChromaDB as the retrieval index over it. That design comes from understanding all three types and why each one alone is insufficient.
