# Lab - Ship a Codex-Assisted Repo Improvement

<LabBrief title="Codex-Assisted Repo Improvement" time="2-3 hours">

#### Objective

Use Codex to make one small, reviewable improvement to a real repository while preserving human control of scope, permissions, verification, and final review.

#### Setup

Install and authenticate Codex using the current official setup path for your machine. Then choose a repository you are allowed to edit.

Before starting:

```bash
git status --short
git switch -c codex-small-improvement
```

#### Deliverables

- A Git diff containing one small code or documentation improvement
- `codex-lab-notes.md` with prompts used, decisions made, verification results, and one reflection
- One Codex review pass on the final diff
- One durable repo instruction added to `AGENTS.md` or a short note explaining why no new instruction was appropriate

#### Hard constraints

- Do not use Full Access for this lab
- Do not let Codex read secrets or private production data
- Do not add dependencies unless Codex stops and explains why first
- Do not merge without human review
- The final diff must be small enough to review in under 10 minutes

</LabBrief>

## Step 1: Read-only exploration

Start by asking Codex to understand a narrow area. Do not edit yet.

```text
I want to make one small, reviewable improvement in this repo.

First, inspect the project structure and identify three candidate improvements that:
- are small enough to review in under 10 minutes
- can be verified with an existing command
- do not require new dependencies
- do not touch secrets, auth policy, billing, or production data

Do not edit files yet. Return:
1. Candidate task
2. Files likely involved
3. Verification command
4. Risk level
```

Choose one candidate. If Codex proposes something broad, push back.

```text
Make the task smaller. I want one behavior or documentation improvement, not a refactor.
```

## Step 2: Write the task spec

Create `codex-lab-notes.md` and record your chosen task:

```md
# Codex lab notes

## Chosen task

## Why this is small enough

## Files in scope

## Non-goals

## Verification command

## Prompts used

## Final review notes

## Reflection
```

Then give Codex the implementation prompt:

```text
Implement the chosen task.

Scope:
- [name the files or directory]

Non-goals:
- Do not refactor unrelated code.
- Do not add dependencies.
- Do not change public behavior outside this task.

Definition of done:
- Run [verification command].
- Summarize changed files.
- Name any remaining test gap.
```

## Step 3: Verify with a real command

Codex should run the command if permissions allow. If not, you run it and paste the output back.

Good verification commands:

```bash
npm run lint
npm run build
pytest
python -m pytest
pnpm test
```

If the command fails, use a repair loop:

```text
The verification command failed.

Command:
[paste command]

Output:
[paste relevant output]

Diagnose the root cause before editing. Make the smallest fix and rerun the same command. If the same error appears twice, stop and explain the blocker.
```

<Callout type="warning">
Do not switch verification commands just to get a pass. If the chosen command is the right signal, keep using it until it passes or you can explain why the repo is already broken.
</Callout>

## Step 4: Ask for a review

After implementation and verification, ask Codex for a separate review:

```text
Review the current diff as if you are blocking a PR.

Findings first, ordered by severity.
Prioritize bugs, regressions, security, and missing tests.
If there are no blocking findings, say that clearly and list residual risk.
Do not edit files during the review.
```

If the review finds a real issue, ask Codex to fix only that issue.

```text
Fix only finding #1. Do not change unrelated files. Rerun the verification command after the fix.
```

## Step 5: Capture durable learning

Look back at the session. Did Codex need a correction that future sessions should inherit?

Good `AGENTS.md` updates:

```md
- For content changes, edit files under `content/modules/` only. Do not change app routing unless explicitly asked.
- Run `npm run lint` after touching `.tsx` files.
- Do not add dependencies for icon, formatting, or utility functions without asking first.
```

Bad `AGENTS.md` updates:

```md
- Be smarter.
- Write better code.
- Always do the right thing.
```

If no durable rule emerged, write this in `codex-lab-notes.md`:

```md
No AGENTS.md update: the correction was task-specific and not likely to recur.
```

## What to submit

Submit the branch or patch with:

- Final diff
- Verification output
- Codex review result
- `codex-lab-notes.md`
- Any `AGENTS.md` update

<Solution label="Show a worked example session">

Every run of this lab is different — this shows one complete session at the quality the rubric calls "Exceeds", so you can compare the *artifacts*, not the steps.

#### The filled-in codex-lab-notes.md

```md
# Codex lab notes

## Chosen task
Add input validation to the CLI's `--port` flag: non-numeric and
out-of-range values currently crash with a TypeError three calls deep.

## Why this is small enough
One function (cli.py:parse_args), one new test file, ~25 changed lines.
Candidate 2 (renaming the config module) touched 14 files — rejected.

## Files in scope
src/cli.py, tests/test_cli.py

## Non-goals
No argparse → click migration (Codex proposed it; declined — refactor,
not improvement). No validation of other flags.

## Verification command
python -m pytest tests/test_cli.py -q

## Prompts used
[the three prompts from steps 1, 2, 4, pasted verbatim]

## Final review notes
Review pass found 1 finding: my error message leaked the raw input
unescaped into a f-string destined for the shell — cosmetic here, but
fixed by repr(). Assessed as valid; fixed; tests rerun: 6 passed.

## Reflection
Codex's first diff also "improved" two adjacent help strings. The
non-goals line stopped a refactor but not micro-edits — next time I'll
add "do not modify lines outside the named function" for surgical tasks.
```

#### What the durable-learning decision looked like

The session produced one correction worth keeping (Codex tried to add `click` as a dependency in step 1's candidate list), so `AGENTS.md` gained:

```md
- Do not propose new dependencies for CLI, formatting, or utility tasks;
  the standard library versions are deliberate.
```

Note what *didn't* go in: "don't edit adjacent help strings" — that correction was task-specific (sometimes you want adjacent cleanups), so it stayed in the notes as a prompting lesson instead. That's the judgment the last rubric criterion is testing: durable rules are about the *repo's* invariants, not about one session's preferences.

#### The two most common ways this lab goes sideways

1. **The diff grows during the repair loop.** The verification fails, and the "smallest fix" touches a second module, which pulls in a third. The fix is in the repair prompt's last line — "if the same error appears twice, stop and explain" — actually stopping. A blocked lab with a clear explanation scores better on Verification than a green check earned by scope creep.
2. **The review pass is asked to "check the code" instead of to block.** Phrasing matters: "review as if blocking a PR, findings first" produces findings; "does this look good?" produces agreement. If your review pass found nothing and cleared nothing explicitly, rerun it with the adversarial phrasing before submitting.

</Solution>
