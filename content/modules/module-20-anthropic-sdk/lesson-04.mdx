# Multi-Agent Patterns with the Raw SDK

In Module 06 you learned what multi-agent systems are conceptually. This lesson shows you how to implement three core patterns â€” sequential, parallel, and critic-loop â€” using only the `anthropic` package and Python's standard library. No framework required.

## The core insight: a worker is a function call

When you use `messages.create()` inside a tool function, you get a worker agent. The orchestrator calls the tool, the tool calls `messages.create()`, and the result comes back as a tool result. The orchestrator has no idea â€” nor does it need to â€” that the tool made an LLM call.

```python
import anthropic
import json

client = anthropic.Anthropic()

def summarise_worker(text: str, max_words: int = 100) -> dict:
    """A worker that summarises text. Runs its own LLM call."""
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",   # cheaper model for simple sub-tasks
        max_tokens=512,
        system="You are a precise summariser. Return only the summary, no preamble.",
        messages=[{"role": "user", "content": f"Summarise in â‰¤{max_words} words:\n\n{text}"}]
    )
    return {"summary": response.content[0].text, "words": len(response.content[0].text.split())}
```

Notice that `summarise_worker` has its own `messages` list â€” completely independent of the orchestrator's history. Context is not shared automatically.

<Callout type="insight">
Each worker call is a separate, independent API call. The model powering the worker has no awareness of the orchestrator's conversation. If a worker needs context from the orchestrator, you must pass it explicitly as part of the worker's system prompt or user message. Design your orchestrator to extract and pass what each worker needs â€” no more, no less.
</Callout>

## Pattern 1: Sequential pipeline

The simplest multi-agent pattern. The orchestrator has tools that call workers in sequence, feeding the output of one into the input of the next.

```python
# Tools that call workers
summarise_tool_schema = {
    "name": "summarise",
    "description": "Summarise a piece of text into a concise paragraph.",
    "input_schema": {
        "type": "object",
        "properties": {
            "text": {"type": "string", "description": "Text to summarise"},
            "max_words": {"type": "integer", "description": "Maximum word count", "default": 100}
        },
        "required": ["text"]
    }
}

translate_tool_schema = {
    "name": "translate",
    "description": "Translate text into a target language.",
    "input_schema": {
        "type": "object",
        "properties": {
            "text": {"type": "string", "description": "Text to translate"},
            "target_language": {"type": "string", "description": "e.g. 'Spanish', 'French', 'Japanese'"}
        },
        "required": ["text", "target_language"]
    }
}

def translate_worker(text: str, target_language: str) -> dict:
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=1024,
        messages=[{"role": "user", "content": f"Translate to {target_language}:\n\n{text}"}]
    )
    return {"translation": response.content[0].text}

TOOL_MAP = {
    "summarise": summarise_worker,
    "translate": translate_worker,
}

# The orchestrator runs a standard agent loop (from Lesson 2)
# It will naturally call summarise first, then translate â€” guided by its system prompt
orchestrator_response = run_agent(
    user_question="Summarise the following article and translate the summary to Spanish: [article text here]",
    tools=[summarise_tool_schema, translate_tool_schema],
    tool_map=TOOL_MAP,
    system="You are a research pipeline. When asked to summarise and translate, always summarise first, then translate the summary."
)
```

The orchestrator decides the order â€” but you can enforce order with the system prompt. Workers are just functions as far as the orchestrator is concerned.

## Pattern 2: Parallel workers with AsyncAnthropic

For independent sub-tasks (e.g., analysing three documents simultaneously), you want to run workers in parallel rather than sequentially. This requires the async client.

```python
import asyncio
import anthropic

# Use AsyncAnthropic for async calls
async_client = anthropic.AsyncAnthropic()


async def analyse_document(document: str, question: str) -> str:
    """Async worker â€” makes a non-blocking API call."""
    response = await async_client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        messages=[{"role": "user", "content": f"Document:\n{document}\n\nQuestion: {question}"}]
    )
    return response.content[0].text


async def parallel_analysis(documents: list[str], question: str) -> list[str]:
    """Run the same question against multiple documents in parallel."""
    tasks = [analyse_document(doc, question) for doc in documents]
    results = await asyncio.gather(*tasks)
    return list(results)


# Run it
documents = ["Document A text...", "Document B text...", "Document C text..."]
results = asyncio.run(parallel_analysis(documents, "What is the main risk mentioned?"))

for i, result in enumerate(results):
    print(f"Document {i+1}: {result}\n")
```

`asyncio.gather()` fires all worker calls simultaneously. Instead of 3 Ã— 2s = 6s wait time, you get roughly max(2s, 2s, 2s) = 2s â€” three times faster.

### Async orchestrator with parallel tool execution

In a real async agent loop, you want to execute all tool calls from a single response in parallel:

```python
async def run_async_agent(user_question: str) -> str:
    messages = [{"role": "user", "content": user_question}]
    MAX_STEPS = 10

    for _ in range(MAX_STEPS):
        response = await async_client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=TOOLS,
            messages=messages,
        )

        if response.stop_reason == "end_turn":
            text_blocks = [b for b in response.content if b.type == "text"]
            return text_blocks[0].text if text_blocks else ""

        messages.append({"role": "assistant", "content": response.content})

        # Execute ALL tool calls in parallel
        tool_blocks = [b for b in response.content if b.type == "tool_use"]
        tasks = [execute_tool_async(block) for block in tool_blocks]
        results = await asyncio.gather(*tasks, return_exceptions=True)

        tool_results = []
        for block, result in zip(tool_blocks, results):
            if isinstance(result, Exception):
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": json.dumps({"error": str(result)}),
                    "is_error": True,
                })
            else:
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": json.dumps(result),
                })

        messages.append({"role": "user", "content": tool_results})

    raise RuntimeError("Agent did not converge")


async def execute_tool_async(block) -> dict:
    """Execute a single tool asynchronously."""
    if block.name not in ASYNC_TOOL_MAP:
        return {"error": f"Unknown tool: {block.name}"}
    return await ASYNC_TOOL_MAP[block.name](**block.input)
```

## Pattern 3: Critic loop

The critic loop improves output quality by having a separate model review and suggest revisions. The orchestrator runs three agents in sequence: drafter â†’ critic â†’ reviser.

```python
def drafter_worker(task: str) -> str:
    """Produces a first draft."""
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=1024,
        messages=[{"role": "user", "content": f"Write a first draft for: {task}"}]
    )
    return response.content[0].text


def critic_worker(draft: str, criteria: str) -> str:
    """Reviews the draft and returns structured feedback."""
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system="You are a strict critic. Return JSON: {score: 1-10, issues: [str], suggestions: [str]}",
        messages=[
            {"role": "user", "content": f"Criteria:\n{criteria}\n\nDraft:\n{draft}"},
            {"role": "assistant", "content": "{"},  # prefill forces JSON
        ]
    )
    return "{" + response.content[0].text


def reviser_worker(draft: str, feedback: str) -> str:
    """Revises the draft based on critic feedback."""
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="You are a precise editor. Apply the feedback and return only the revised text.",
        messages=[{"role": "user", "content": f"Original:\n{draft}\n\nFeedback:\n{feedback}\n\nRevise it."}]
    )
    return response.content[0].text


def critic_loop(task: str, criteria: str, max_rounds: int = 3) -> str:
    draft = drafter_worker(task)
    
    for round_num in range(max_rounds):
        feedback_json = critic_worker(draft, criteria)
        feedback = json.loads(feedback_json)
        
        print(f"Round {round_num + 1}: score={feedback['score']}, issues={len(feedback['issues'])}")
        
        if feedback["score"] >= 8:
            print(f"Accepted after {round_num + 1} round(s)")
            return draft
        
        draft = reviser_worker(draft, json.dumps(feedback))
    
    print(f"Returning best draft after {max_rounds} rounds")
    return draft


# Run it
result = critic_loop(
    task="Write a one-paragraph explanation of transformer attention for a high school student",
    criteria="Must use no jargon, must include an analogy, must be under 100 words"
)
print(result)
```

## When to use raw SDK vs LangGraph

The raw SDK is the right tool in specific situations. Knowing when to stop and reach for a framework saves engineering time.

| Situation | Raw SDK | LangGraph |
|---|---|---|
| Simple linear pipeline | Best choice | Unnecessary overhead |
| Need checkpointing / resume | Not built-in | Built-in |
| Conditional branching with state | Manageable by hand | Purpose-built |
| Human-in-the-loop interrupts | Manual implementation | `interrupt_before` |
| Parallel fan-out | `asyncio.gather` | Built-in |
| Debugging a framework | Essential | â€” |

The rule of thumb: if you can describe the agent's control flow in a linear sequence of steps with no branching state machine, the raw SDK is simpler. Once you need a stateful graph with conditional edges, checkpointing, and interrupts, LangGraph earns its weight.

<Callout type="insight">
The raw SDK patterns in this lesson map directly onto what LangGraph does internally. A LangGraph node is a function that receives state and returns updated state â€” equivalent to a worker function. A LangGraph edge is a conditional check on state â€” equivalent to an `if response.stop_reason == ...` check. Understanding the raw SDK makes LangGraph transparent.
</Callout>
