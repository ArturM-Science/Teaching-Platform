# The API Contract: Messages, Roles, and Content Blocks

Every call to Claude — whether you use the raw SDK, LangGraph, or CrewAI — eventually becomes a single HTTP POST to the Messages API. This lesson teaches you that API in full. Once you understand the exact shape of the request and the response, every framework becomes readable, every bug becomes findable.

## Installation and setup

```bash
pip install anthropic
```

```python
import anthropic

client = anthropic.Anthropic()  # reads ANTHROPIC_API_KEY from environment
```

The `Anthropic()` constructor looks for your API key in the environment variable `ANTHROPIC_API_KEY`. You can also pass it explicitly with `api_key="sk-ant-..."`, but using environment variables is the right default for any code that might touch version control.

<Callout type="warning">
`max_tokens` is required in every `messages.create()` call. Omitting it raises a validation error immediately — Claude has no default. This differs from some other SDKs. Always set it, even if you set it high.
</Callout>

## Your first call

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "What is 2 + 2?"}
    ]
)

print(response.content[0].text)
# Output: 4
```

That is the minimum viable call. Now let's understand every piece of it.

## Anatomy of `messages.create()`

The full signature looks like this:

```python
response = client.messages.create(
    model="claude-sonnet-4-6",       # required: model string
    max_tokens=1024,                  # required: cap on output tokens
    system="You are a helpful...",    # optional: system prompt (string)
    messages=[                        # required: conversation history
        {"role": "user",    "content": "..."},
        {"role": "assistant","content": "..."},
        {"role": "user",    "content": "..."},
    ],
    tools=[...],                      # optional: tool schemas
    temperature=1.0,                  # optional: 0.0–1.0, default 1.0
    stop_sequences=["DONE"],          # optional: custom stop strings
)
```

### The `messages` array

The messages array represents the conversation history. Each entry is a dict with two keys: `role` and `content`.

**Valid roles in the messages array:** `"user"` and `"assistant"` only.

There is no `"system"` role in the messages array. The system prompt is a separate top-level parameter named `system`. This is a frequent source of confusion for developers who come from OpenAI's API, where `{"role": "system", ...}` is a messages-array entry.

<Callout type="warning">
**Common mistake:** placing the system prompt inside the messages array as `{"role": "system", "content": "..."}`. The Anthropic API will return a `400 Bad Request` — `"system"` is not a valid role value. Always pass the system prompt as the `system=` parameter.
</Callout>

### Content: string or list of blocks

The `content` value of each message can be either a plain string (shorthand) or a list of **content blocks** (full form):

```python
# Shorthand — fine for simple text
{"role": "user", "content": "What's the weather like?"}

# Full form — required when mixing content types
{"role": "user", "content": [
    {"type": "text", "text": "Describe this image:"},
    {"type": "image", "source": {"type": "url", "url": "https://..."}},
]}
```

The API always returns content as a list of blocks, even if you sent a plain string. This is why `response.content[0].text` is the pattern to extract the first text block from a response.

## Content block types

There are four content block types you need to know. Together they cover everything the API can send or receive.

### 1. Text block

```python
{"type": "text", "text": "Hello, world!"}
```

Used in both user messages and assistant responses. The most common type.

### 2. Image block

```python
{
    "type": "image",
    "source": {
        "type": "base64",
        "media_type": "image/jpeg",  # or image/png, image/gif, image/webp
        "data": "<base64-encoded bytes>",
    }
}
```

Image blocks can only appear in `user` messages. Vision is covered in depth in Lesson 3.

### 3. Tool use block

```python
{
    "type": "tool_use",
    "id": "toolu_01XFDUDYJgAACTvnkyLAND",
    "name": "get_weather",
    "input": {"location": "London", "unit": "celsius"}
}
```

This block appears in **assistant** messages when the model decides to call a tool. You never write this block yourself — the API produces it. Your job is to detect it in the response and act on it. Tool use is covered in depth in Lesson 2.

### 4. Tool result block

```python
{
    "type": "tool_result",
    "tool_use_id": "toolu_01XFDUDYJgAACTvnkyLAND",
    "content": '{"temperature": 14, "condition": "cloudy"}'
}
```

This block appears in **user** messages when you feed a tool's output back to the model. You write this block yourself, after executing the tool. The `tool_use_id` must match the `id` from the corresponding `tool_use` block — this is how the model knows which tool call produced which result. Tool use is covered in depth in Lesson 2.

## The response object

```python
response = client.messages.create(...)

response.id              # "msg_01XFDUDYJgAACTvnkyLAND" — unique message ID
response.model           # "claude-sonnet-4-6" — model that was used
response.role            # "assistant" — always
response.content         # list of content blocks
response.stop_reason     # "end_turn" | "tool_use" | "max_tokens" | "stop_sequence"
response.usage.input_tokens    # tokens consumed by your input
response.usage.output_tokens   # tokens consumed by the response
```

### stop_reason values

| Value | Meaning |
|---|---|
| `"end_turn"` | Model finished normally. Read `response.content` for the answer. |
| `"tool_use"` | Model wants to call a tool. Extract the `tool_use` blocks and execute them. |
| `"max_tokens"` | Response was truncated at your `max_tokens` limit. Increase it or redesign the prompt. |
| `"stop_sequence"` | Model generated one of your custom `stop_sequences`. |

The `stop_reason` is your primary dispatch signal in an agent loop. Check it before deciding what to do next.

## A more complete example

Here is a multi-turn conversation to show the messages array in action:

```python
import anthropic

client = anthropic.Anthropic()

messages = [
    {"role": "user", "content": "My name is Ada."},
    {"role": "assistant", "content": "Hello Ada! How can I help you today?"},
    {"role": "user", "content": "What's my name?"},
]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=256,
    system="You are a helpful assistant with a good memory.",
    messages=messages,
)

print(response.content[0].text)
# Output: Your name is Ada.
print(response.stop_reason)
# Output: end_turn
print(f"Used {response.usage.input_tokens} input tokens, {response.usage.output_tokens} output tokens")
```

The model has access to the full messages history. You maintain that history in your application. The API is stateless — every call sends the entire conversation from scratch.

<Callout type="insight">
The API is completely stateless. The model has no memory between calls unless you include prior turns in the messages array. This is both a constraint and a feature: you have full control over what the model remembers. In a production system, you manage conversation history in your own data store.
</Callout>

## Extracting content safely

Because `response.content` is a list and may contain multiple blocks (especially when tools are involved), it is good practice to be explicit about which block you want:

```python
# Fragile — assumes content[0] is always a text block
text = response.content[0].text

# Safer — find the first text block explicitly
text_blocks = [b for b in response.content if b.type == "text"]
text = text_blocks[0].text if text_blocks else ""
```
