# Lab — Design a Multi-Agent Research Team

<LabBrief
  title="Design a Multi-Agent Research Team"
  duration="~45 minutes"
  objective="Design a four-agent pipeline that answers a complex research question. You will write system prompts for each agent, choose and justify a communication pattern, identify failure modes, and write the orchestrator's synthesis prompt."
/>

## The scenario

Your team is building an AI research assistant. A user submits the question:

> "What is the current state of large language model efficiency research — specifically model compression and inference optimisation — and how do the leading approaches compare?"

This question is too broad for a single agent to answer well. It requires gathering information from multiple sub-domains, comparing findings across sources, and producing a coherent synthesis. Your job is to design the multi-agent pipeline that handles it.

## The target architecture

Your pipeline has four agents:

```
User question
      ↓
[Orchestrator]
      ↓ decomposes into 2 sub-questions
      ↓
[Specialist A]          [Specialist B]
Model compression       Inference optimisation
      ↓                       ↓
      ↓ ← outputs fed back →  ↓
[Critic / Reviewer]
Reviews combined output
      ↓
[Orchestrator]
Final synthesis
      ↓
Answer to user
```

## Task 1 — Write system prompts for each agent

Write a system prompt (4–8 sentences) for each of the four agents. Each system prompt must specify:

1. The agent's role and what it does
2. What input it receives and in what format
3. What output it must produce and in what format
4. What it explicitly does NOT do (scope boundary)

**Orchestrator system prompt:**

```
You are the orchestrator of a multi-agent research pipeline...
[Write your prompt here]
```

**Specialist A (model compression) system prompt:**

```
You are a specialist researcher focused exclusively on model compression techniques...
[Write your prompt here]
```

**Specialist B (inference optimisation) system prompt:**

```
You are a specialist researcher focused exclusively on inference optimisation...
[Write your prompt here]
```

**Critic / Reviewer system prompt:**

```
You are a critic agent. You do not generate research — you evaluate it...
[Write your prompt here]
```

<Callout type="tip">
Good system prompts for specialists include an explicit refusal: "If asked about topics outside model compression, return: out_of_scope: true." Scope boundaries prevent specialists from drifting into each other's domains.
</Callout>

## Task 2 — Choose and justify a communication pattern

From the three patterns covered in Lesson 3, choose one (or a justified hybrid) for your pipeline.

Answer these questions:

1. Which pattern did you choose?
2. Why does it fit this pipeline better than the alternatives?
3. What is the main failure risk of your chosen pattern in this specific pipeline, and how would you mitigate it?

Consider: does this pipeline benefit from parallel execution (Specialist A and B running simultaneously)? Does it require strict auditability? Do agents need to build incrementally on each other's partial results?

## Task 3 — Identify two failure modes and add mitigations

From the MAST taxonomy, identify two failure modes your pipeline is vulnerable to. For each:

1. Name the failure mode (e.g., "memory poisoning", "coordination deadlock", "format mismatch")
2. Describe specifically how it could occur in your pipeline
3. Describe the mitigation you would add

**Example format:**

| Failure mode | How it occurs in this pipeline | Mitigation |
|---|---|---|
| Memory poisoning | Specialist A hallucinates a compression ratio and... | Structured output schema with `source_url` field required... |
| Format mismatch | Specialist B returns markdown; Critic expects JSON... | Output contract defined in system prompt: all agents return... |

## Task 4 — Write the orchestrator's synthesis prompt

Write the prompt the orchestrator uses to synthesise the two specialists' outputs into a final answer. The synthesis prompt should:

1. Accept two structured inputs (Specialist A output, Specialist B output) and the Critic's verdict
2. Instruct the orchestrator to note where the two specialists agree and where they diverge
3. Require the orchestrator to flag any claims the Critic marked as unverified
4. Produce a final answer in a defined format (you choose the format)

```
[Synthesis prompt]
You have received research from two specialists and a quality verdict from the Critic.
Specialist A output: {specialist_a_output}
Specialist B output: {specialist_b_output}
Critic verdict: {critic_verdict}

Your task:
[Write your synthesis instructions here]
```

<Solution label="Show a worked example design">

This is one defensible design, not the only one. Compare it to yours task by task — if you made different choices with reasons, that can score just as well.

#### Task 1 — System prompts

**Orchestrator:**

```text
You are the orchestrator of a multi-agent research pipeline. You receive one
research question from the user. Your first job is to decompose it into exactly
two non-overlapping sub-questions: one on model compression, one on inference
optimisation, each as JSON: {"sub_question": "...", "assigned_to": "specialist_a|b"}.
Your second job, after the Critic's verdict arrives, is to synthesise the
specialists' outputs into the final answer using your synthesis instructions.
You do NOT research topics yourself, and you do NOT edit specialist findings —
you only decompose, route, and synthesise.
```

**Specialist A (model compression):**

```text
You are a specialist researcher focused exclusively on model compression
techniques: quantisation, pruning, distillation, and low-rank factorisation.
You receive one sub-question as JSON with a "sub_question" field. You return
JSON: {"findings": [{"claim": "...", "technique": "...", "evidence": "...",
"confidence": "high|medium|low"}], "summary": "..."}. Every claim must carry
an evidence string and a confidence level. If the sub-question is outside
model compression, return {"out_of_scope": true} and nothing else. You do
NOT cover inference optimisation, serving infrastructure, or hardware.
```

**Specialist B (inference optimisation):** identical structure, scoped to KV-cache management, speculative decoding, batching, and attention optimisations, with the mirror-image refusal.

**Critic:**

```text
You are a critic agent. You do not generate research — you evaluate it. You
receive both specialists' JSON outputs. For each claim, check: is it supported
by its evidence string, is the confidence level justified, and does it
contradict any claim from the other specialist? You return JSON:
{"verdict": "approve|revise", "unverified_claims": ["..."],
"contradictions": ["..."], "notes": "..."}. You do NOT rewrite findings,
add new claims, or soften your verdict to be agreeable.
```

#### Task 2 — Communication pattern

**Choice: direct handoff (orchestrator-mediated), with the two specialists running in parallel.**

Why it beats the alternatives here: the pipeline is a fixed diamond — decompose, fan out to exactly two known specialists, fan in, review, synthesise. A shared message queue buys nothing because no agent subscribes dynamically; a blackboard adds the risk that one specialist reads the other's partial findings and anchors on them, which this design explicitly wants to avoid (independent findings make divergence informative at synthesis time). Parallelism is preserved because the handoff is orchestrator→both simultaneously.

Main risk of this choice: the orchestrator is a single point of failure and a bandwidth bottleneck — if its decomposition is bad, both specialists do the wrong work and nothing downstream can detect it. Mitigation: the decomposition is itself JSON with a fixed schema, validated in code before dispatch, and the Critic's contradiction check provides a late tripwire for incoherent sub-questions.

#### Task 3 — Failure modes (MAST)

| Failure mode | How it occurs in this pipeline | Mitigation |
|---|---|---|
| Information withholding / lossy handoff | Specialist A's `summary` field is what the orchestrator reads at synthesis; nuance in individual findings (a "low" confidence flag on a headline number) is dropped, so the final answer states the number as fact | Synthesis prompt receives the full `findings` array, not just summaries, and is required to propagate the lowest confidence level attached to any claim it uses |
| Verifier capitulation / weak critic | The Critic returns "approve" on everything because nothing forces specificity — approval is the path of least resistance | Critic schema requires `unverified_claims` and `contradictions` arrays even when empty, plus orchestrator-side rule: an approve verdict with empty notes on findings containing any "low" confidence claim triggers one re-review |

#### Task 4 — Synthesis prompt

```text
You have received research from two specialists and a quality verdict from the Critic.
Specialist A output: {specialist_a_output}
Specialist B output: {specialist_b_output}
Critic verdict: {critic_verdict}

Your task:
1. Produce a final answer with exactly these sections:
   ## Where the approaches agree
   ## Where they diverge or trade off
   ## Open questions and unverified claims
   ## Bottom line (max 4 sentences)
2. In the first two sections, draw only on claims present in the specialists'
   findings arrays. Carry each claim's confidence level inline, e.g. "(confidence: medium)".
3. Every claim listed in the Critic's unverified_claims must appear in the
   "Open questions" section, marked "unverified" — it may not appear anywhere else.
4. If the Critic's verdict is "revise", do not synthesise: return
   {"action": "revision_needed", "reason": <critic notes>} instead.
```
