# Lesson 2 — Agent Roles: Orchestrator, Subagent, Specialist, Critic

Every production multi-agent system organises around the same four functional roles, regardless of framework. The names vary — supervisor, worker, lead, reviewer — but the functions are consistent. Understanding these roles before writing any code prevents the most common design mistake: giving agents responsibilities that belong to a different role.

## The four roles

### Orchestrator

The orchestrator plans the overall task, decomposes it into subtasks, delegates to subagents, tracks completion, and synthesises the results. It does not do ground-level work — its value is coordination, not execution.

Anthropic's Research product is the canonical public example. The lead agent analyses the user's request, develops a strategic research approach, and spawns three to five specialised subagents with specific research mandates. After the subagents return, the orchestrator runs a dedicated citation-verification pass before synthesising the final answer. This deliberate separation of synthesis from verification is evidence of role discipline.

The orchestrator holds the most critical context. If it fails, the entire pipeline stalls — it owns the synthesis state. Design it to be resilient: save research plans to external memory before context windows fill, and define explicit escalation paths for subagent failures.

### Subagent

A subagent receives a specific, bounded task from the orchestrator, executes it using tools, and returns a structured result. It operates with its own isolated context window, which is what enables genuine parallelism.

The subagent's return contract matters as much as the task itself. It should return a summary relevant to the orchestrator's synthesis need, not its full context. The orchestrator's context window fills quickly if every subagent returns everything it processed.

In the Claude Agent SDK, subagent outputs are tracked via a `parent_tool_use_id` field — each message from within a subagent's context can be traced back to the orchestrator call that spawned it.

### Specialist

A specialist is a subagent with intentional constraints. Where a general subagent does whatever the task requires, a specialist is designed to refuse work outside its domain.

A software review pipeline might include an architecture specialist, an implementation specialist, a test-coverage specialist, and a security specialist. Each specialist's system prompt explicitly defines what it evaluates — and what it does not. The security specialist does not comment on architecture. The architecture specialist does not run test analysis.

This constraint is a feature, not a limitation. Focused system prompts with narrow scope produce higher-quality outputs than general prompts that try to cover everything. A specialist failure affects only its domain — the architecture review continues even if the security scan fails.

### Critic / Reviewer

The critic receives another agent's output and evaluates it against defined criteria. It returns a structured verdict: pass or fail, with specific gaps and suggested corrections. It does not modify outputs — it evaluates them.

The critic is the primary mechanism for inter-agent quality assurance. Anthropic's research on multi-agent systems found that including a dedicated reviewer "significantly reduces hallucinations and improves overall reliability" compared to systems where the generating agent also evaluates its own output.

One important caveat: a critic that consistently passes bad output is worse than no critic at all — it creates false confidence. Test your critic against known-bad outputs before deploying.

## Roles are functional, not structural

The same LLM can play different roles in sequence. An agent that synthesises research (orchestrator role) might also verify citations (critic role) in the same pipeline. What matters is that each function has a clear definition of what it receives, what it produces, and what happens when it fails.

A common mistake is assigning multiple conflicting roles to a single agent. An orchestrator that also tries to do specialist research loses focus on both. Keep roles separated — if an agent is trying to do two things, it is usually doing both worse than two dedicated agents would.

## The performance data

Anthropic's internal multi-agent research system — orchestrator plus three to five subagents plus citation verification — outperformed single-agent Claude Opus 4 by 90.2% on internal evaluations. The cost: approximately 15× the tokens of a normal chat interaction. Token usage alone explained 80% of the variance in research quality.

This is the trade-off: multi-agent with well-defined roles produces significantly better output for complex tasks, at significantly higher cost. Role definition is what makes the cost worthwhile.
