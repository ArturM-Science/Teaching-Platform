# Lesson 1 — What Makes a System Multi-Agent?

The word "multi-agent" gets applied to a lot of things. Parallel steps, branching workflows, systems that call an LLM twice — all of these get called multi-agent. Most of them are not. The distinction matters because the two architectures have fundamentally different failure modes, cost profiles, and design requirements.

## The architectural line

Anthropic's own documentation draws the line clearly:

- **Workflows** are systems where LLMs and tools are orchestrated through predefined code paths.
- **Agents** are systems where LLMs dynamically direct their own processes and tool usage, maintaining control over how they accomplish tasks.

A multi-agent system is a system where multiple agents — each with its own context window and decision-making — coordinate dynamically to complete a task. The coordination itself is not predefined; the agents determine at runtime what to do next.

A parallel workflow, by contrast, is a predefined structure that happens to run concurrently. The branching points are fixed in code. There is no agent deciding whether to spawn a new worker or re-plan the approach.

## The four architectural differences

| Dimension | Parallel workflow | Multi-agent system |
|---|---|---|
| **Control flow** | Predetermined by the developer | Determined dynamically by agents |
| **Coordination** | Implicit in code structure | Explicit agent-to-agent communication |
| **Context** | Shared or split, but fixed | Each agent has its own isolated context window |
| **Adaptation** | Cannot adapt to unexpected branches | Can spawn new subagents, re-plan, reroute |

The context window difference is the most practically important. In a parallel workflow, you might split a document across two prompt calls — but the structure of that split is decided before runtime. In a multi-agent system, the orchestrating agent decides mid-execution whether to spawn two or five subagents, what each one should focus on, and how to handle an unexpected result.

## When multi-agent is the right choice

Three conditions justify multi-agent complexity:

1. **The task decomposes by domain.** A security review, a legal review, and a correctness review require different expertise and benefit from separate focused system prompts.

2. **Information volume exceeds one context window.** If you need to process 50 documents and synthesise findings, a single agent cannot hold all the documents at once. Subagents each process a slice.

3. **You need dynamic replanning.** If discovering that Institution A has no public data should change how you approach Institution B, the orchestrator needs to be able to adapt — not just run the predefined fallback branch.

## The cost of coordination

Multi-agent complexity is not free. A 2024 Google research study found that coordination overhead reduces sequential reasoning performance by 39–70% compared to single-agent approaches. The orchestrator has to manage state, interpret subagent outputs, and synthesise — all of which adds latency and token cost.

Anthropic's guidance: start with the simplest architecture. "Optimising single LLM calls with retrieval and in-context examples is usually enough" for most applications. Add multi-agent structure only when there is a measurable benefit.

The most common mistake is reaching for multi-agent architecture because a task has multiple steps. Multiple steps is a workflow. Multi-agent is for tasks where the coordination itself must be dynamic.
