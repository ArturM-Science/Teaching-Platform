# Lesson 3 · The Economics of LLM APIs

Most engineering teams add cost monitoring after they get a surprise bill. This lesson helps you avoid that pattern by understanding token economics before you scale, and by giving you concrete optimisations you can apply today.

## Input vs output: why output costs more

Every major frontier model prices input and output tokens separately, with output consistently 4–6× more expensive:

| Model | Input (per 1M) | Output (per 1M) | Ratio |
|---|---|---|---|
| GPT-4o | $2.50 | $10.00 | 4× |
| GPT-4o-mini | $0.15 | $0.60 | 4× |
| Claude Sonnet 4.6 | $3.00 | $15.00 | 5× |
| Claude Haiku 4.5 | $1.00 | $5.00 | 5× |
| Gemini 1.5 Flash | $0.075 | $0.30 | 4× |

The reason: input tokens are processed in parallel (one forward pass reads all input simultaneously), while output tokens are generated autoregressively — each token requires a full forward pass that depends on all previous tokens. More compute per output token → higher price.

The implication: verbose completions are expensive. Chain-of-thought reasoning in the completion costs 5× more per token than the same tokens in the input. If you have a reasoning model generating long scratchpads, track that output token count carefully.

## Where cost actually goes in a RAG agent call

A typical RAG customer support query:

```
System prompt:       600 tokens  (instructions, persona, safety rules)
User message:        150 tokens  (the question)
Retrieved context:   750 tokens  (3 chunks × 250 tokens)
Total input:       1,500 tokens

Agent completion:    280 tokens  (the answer)
```

On GPT-4o:
```
Input cost:  1,500 / 1,000,000 × $2.50 = $0.00375
Output cost:   280 / 1,000,000 × $10.00 = $0.00280
Total per query:                          $0.00655
```

At 10,000 queries/day: $65.50/day → **$1,965/month**.

The system prompt (600 tokens) is paid on every single request. If it never changes, you pay for it millions of times. This is the highest-leverage caching target.

## Prompt caching (Anthropic)

Anthropic's prompt caching discounts tokens in a repeated prefix — the system prompt, a large knowledge base, or a long document shared across many requests:

```python
response = client.messages.create(
    model="claude-sonnet-4-5",
    system=[{
        "type": "text",
        "text": SYSTEM_PROMPT,
        "cache_control": {"type": "ephemeral"}  # mark for caching
    }],
    messages=[{"role": "user", "content": user_message}],
    max_tokens=512
)

# Inspect cache usage in the response:
print(response.usage.cache_read_input_tokens)     # tokens served from cache (0.10× price)
print(response.usage.cache_creation_input_tokens)  # tokens written to cache (1.25× price)
```

Cache economics:
- **Minimum eligible block**: 1,024 tokens
- **Cache hit cost**: 10% of normal input token price
- **Cache write cost**: 125% of normal (one-time cost to populate the cache)
- **TTL**: 5 minutes, reset on each hit

For our example: 600-token system prompt cached at 10% → $0.000375 instead of $0.00150 per query — 75% saving on that segment. At 10,000 queries/day that is $11.25/day saved from caching one system prompt.

Google Gemini has a similar mechanism (minimum 32,768 tokens, configurable TTL up to 1 hour) for large context caching.

<Callout type="tip">
The system prompt is the highest-value caching target because it is identical across all requests. After that, consider caching large retrieved documents that appear in many queries — a legal contract or product specification referenced repeatedly is a good candidate.
</Callout>

## Batch API (50% discount, async)

For workloads that do not need real-time responses, the Anthropic Batch API cuts all token prices in half:

```python
response = client.messages.batches.create(
    requests=[
        {
            "custom_id": f"doc_{i}",
            "params": {
                "model": "claude-sonnet-4-5",
                "max_tokens": 1024,
                "messages": [{"role": "user", "content": doc}]
            }
        }
        for i, doc in enumerate(documents)
    ]
)
batch_id = response.id

# Poll for completion (typically 1–4 hours, max 24 hours)
import time
while True:
    batch = client.messages.batches.retrieve(batch_id)
    if batch.processing_status == "ended":
        break
    time.sleep(60)

for result in client.messages.batches.results(batch_id):
    print(result.custom_id, result.result.message.content)
```

Use Batch API for: nightly eval runs, bulk document summarisation, generating training data, offline enrichment pipelines. Do **not** use it for user-facing real-time responses.

## Model routing

Route requests to an appropriate model tier based on task complexity:

```python
ROUTING_PROMPT = (
    "Classify this query as 'simple' (factual lookup, yes/no, status check) "
    "or 'complex' (multi-step reasoning, explanation, comparison, analysis). "
    "Output one word only.\n\nQuery: {query}"
)

def route_and_call(user_message: str) -> str:
    route = openai.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": ROUTING_PROMPT.format(query=user_message)}],
        max_tokens=5
    ).choices[0].message.content.strip().lower()

    model = "gpt-4o" if route == "complex" else "gpt-4o-mini"
    return openai.chat.completions.create(
        model=model,
        messages=[
            {"role": "system",  "content": SYSTEM_PROMPT},
            {"role": "user",    "content": user_message}
        ]
    ).choices[0].message.content
```

If 70% of queries route to GPT-4o-mini and 30% to GPT-4o, the blended cost per query drops from $0.00655 to roughly $0.0027 — a 59% reduction with no change in quality on complex queries. The routing classifier costs roughly $0.0001 per call, which is negligible.

## Semantic caching

Semantic caching stores previous LLM responses indexed by the embedding of the query. On a cache hit (cosine similarity above threshold), return the cached answer without calling the LLM.

Works well: high-traffic FAQ bots, dashboards with a small set of common queries, translation services with repetitive source texts.

Works poorly: agents processing unique user-provided documents, creative writing, search with diverse queries — cache hit rate will be too low to justify the overhead.

## Validate cost cuts against your eval suite

Every optimisation above — model routing, semantic caching, swapping to a cheaper model tier — is also a behaviour change, not just a cost change. "70% of queries route to GPT-4o-mini" is only a win if those queries still pass the same quality bar they did on the bigger model. Don't ship a cost cut on the strength of the dollar savings alone.

Run the eval harness you built in Module 4 against both configurations before and after the change, broken down by category, not as one blended score:

```python
baseline_results = run_eval_suite(model="gpt-4o", eval_cases=eval_suite)
optimised_results = run_eval_suite(model="gpt-4o-mini", eval_cases=eval_suite)

for category in ["happy", "adversarial", "edge", "regression"]:
    base_score = score_by_category(baseline_results, category)
    opt_score = score_by_category(optimised_results, category)
    print(f"{category}: {base_score:.0%} → {opt_score:.0%}")
```

A drop that's concentrated in one category (say, adversarial cases failing more often on the cheaper model) is a different decision than a uniform small drop across all categories — the per-category breakdown tells you which. "Routing 70% of traffic to the smaller model holds 97% of eval performance at 60% of the cost" is a claim you can defend in a review. "It's cheaper and seems fine" is not.

#### What happened
A document-processing agent routed queries using a lightweight classifier: short or structured inputs went to GPT-4o-mini ($0.15/$0.60 per MTok), complex analysis went to GPT-4o ($2.50/$10.00 per MTok). On average, 74% of traffic routed to the cheap tier — blended cost was $0.0027 per query, well within budget.

On a Tuesday afternoon, a customer began uploading malformed PDFs: valid file headers, but body content that was truncated mid-sentence. The routing classifier received these as inputs, timed out waiting for the embedding service it called, caught the exception, and fell through to the default branch — which was GPT-4o. Every malformed document routed to the expensive tier.

The customer uploaded 1,400 documents over four days before anyone noticed. At $0.031 per query on GPT-4o (versus $0.0027 on the mixed tier), cost ran at 11.5× the normal rate. The billing system had one alert: a fixed monthly cap at $6,000 — which had not yet fired.

#### Root cause
Two independent gaps compounded:

1. **No per-routing-tier cost tracking.** The team tracked total daily spend, not spend broken down by routing decision. A dashboard showing "GPT-4o-mini share" would have shown the routing mix drop from 74% to 0% on Tuesday afternoon and triggered an immediate investigation.

2. **Silent failure in the routing path.** The embedding service timeout was caught and swallowed. The fallback was the expensive tier, not an error — so the agent appeared healthy by every standard metric: HTTP 200, correct outputs, no elevated error rate. There was no cost-per-routing-decision metric wired to an alert.

#### How to find it
Add a `routing_decision` tag to every LLM span:

```python
langfuse_context.update_current_observation(
    metadata={"routing_decision": "gpt-4o-mini", "routing_reason": "simple"}
)
```

Then alert when the fraction of `gpt-4o` spans exceeds a threshold — say, >40% over a 30-minute window (versus the normal ~26%). This alert would have fired within 30 minutes of the malformed uploads starting.

The lesson-05 SLO framework covers this under the "Cost per hour" dashboard panel: alert when cost per hour exceeds 2× the 7-day average. That alert would have fired within 2 hours of the incident starting — not on day 4.

#### Fix
Never let a routing failure silently escalate to the expensive tier. Make the fallback explicit:

```python
def route_request(query: str) -> str:
    try:
        tier = classify_tier(query)  # may call embedding service
    except Exception as e:
        log("WARN", "routing_failed", error=str(e))
        tier = "gpt-4o-mini"  # fail to cheap, not expensive
    return tier
```

Add a cost-per-routing-tier metric and alert on routing mix anomalies. A model routing optimisation that cuts cost 59% is worthless if a single upstream timeout can silently undo it.
