# Lesson 8 . Choosing Your First Agent Framework

The best first framework is not LangChain, LangGraph, CrewAI, or any agent SDK. The best first framework is the tiny loop you just wrote yourself in plain Python, because it shows you where the model stops and your program begins. Now that you have built it, you are in a position to judge what the frameworks are actually selling.

## You started with the smallest useful stack

In the lab you used:

- Python 3.11+
- One model API client
- One or two local tools
- A fixed JSON response format
- A hard `max_steps` limit

That was enough to learn the core mechanics without hiding them behind framework vocabulary.

<Callout type="key">
Use a framework when you already understand the loop and need production conveniences. Do not use a framework to learn what an agent is.
</Callout>

## Why you did not start with LangChain or LangGraph

Frameworks are useful, but they compress several decisions into one line of code:

- How tools are described to the model
- How tool calls are parsed
- How tool results are added back to context
- How many times the loop may run
- What happens when a tool fails
- How state is stored between steps

That compression is valuable later. At the beginning, it makes the important parts invisible.

Imagine your agent calls the calculator tool three times and never finishes. Because you built it yourself, you know where to inspect: the stop condition, the action parser, the message history, or the tool result. If a framework had built it for you, the same bug would feel mysterious.

## The architecture you built

Your lab agent was a few small pieces:

```text
agent.py
  run_agent(goal)
  call_model(messages)
  parse_action(text)

tools.py
  calculator(expression)
  lookup(topic)

prompts.py
  SYSTEM_PROMPT
```

The model only decides what action to request. Python decides whether that request is allowed, calls the tool, records the result, and decides whether the loop continues.

## The contract between model and code

Your agent ran on a strict output contract. The simplest version is JSON:

```json
{
  "thought": "I need to calculate the total before answering.",
  "action": "calculator",
  "args": { "expression": "19.99 * 3" }
}
```

To finish:

```json
{
  "thought": "I have enough information to answer.",
  "action": "final",
  "answer": "The total is 59.97."
}
```

This contract was the real "framework" of your first agent. It told the model what shapes it may output and told your code what shapes it must handle.

## When to introduce a real framework

Now that you have built the bare-metal agent, frameworks are easier to evaluate. You can ask what each one buys you:

| Need | Good next tool |
|---|---|
| You want graph-shaped workflows with explicit state | LangGraph |
| You want provider-native tool calling and hosted agent features | OpenAI Agents SDK |
| You want multiple role-based workers with tasks | CrewAI |
| You want a broad ecosystem of chains, retrievers, and integrations | LangChain |

For this course, the first step was plain Python. Module 5 and Module 6 introduce orchestration and multi-agent patterns where frameworks start earning their keep.

<Callout type="tip">
The first agent should feel almost boring: one loop, one parser, one tool table, one stop condition. Boring is good. Boring code is inspectable code.
</Callout>
