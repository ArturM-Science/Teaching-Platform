# Lesson 3 · Determinism, Temperature, and "It Worked Yesterday"

One of the most disorienting things about building with LLMs is that the same input doesn't always produce the same output. This lesson explains why — and what to do about it.

## Temperature and sampling, revisited

From Lesson 1: the model produces a probability distribution and samples from it. Temperature controls the shape of that distribution.

- **Temperature 0**: always pick the highest-probability token. Near-deterministic — same input almost always gives the same output.
- **Temperature 0.7** (typical default): sample proportionally. Outputs vary run to run.
- **Temperature 1.0+**: flatten the distribution further, allowing lower-probability tokens. More creative, more surprising, less reliable.

For agent tasks that need structured output — tool calls, JSON, code — use low temperature. The goal is reliability, not creativity.

## Why "temperature 0" isn't truly deterministic

Even at temperature 0, you can get different outputs. Three reasons:

1. **Model version changes.** Providers silently update their models. `gpt-4o` today is not `gpt-4o` three months ago.
2. **Floating point non-determinism.** GPU parallelism introduces tiny numerical differences that can change which token wins a tie.
3. **System prompt drift.** If your system prompt is dynamic (includes a timestamp, user info, etc.), the context changes even if your task input doesn't.

<Callout type="warning">
"It worked in testing" is not a guarantee. Pin your model version. Make your system prompt deterministic. Log inputs and outputs. You'll thank yourself later.
</Callout>

## What this means for agent reliability

A single non-deterministic step early in an agent run can cascade. If the first tool call parses differently from what you expect, everything downstream is wrong.

The defenses:
- **Structured outputs** — constrain the model's output format so parsing can't fail (Module 1)
- **Schema validation** — use Pydantic to validate what the model returned before acting on it
- **Regression tests** — record known-good runs and test against them (Module 4)
- **Logging** — capture every input/output so you can reproduce failures (Module 9)

## The practical upshot

When a colleague says "it worked yesterday," the diagnosis checklist is:
1. Did the model version change?
2. Did the context change (system prompt, user data)?
3. Was the previous run actually tested, or just run once?

Non-determinism is a fact of the architecture. Your job is to build systems that are robust to it — not to wish it away.
