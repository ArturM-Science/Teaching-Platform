# Mental Model: Roles, Tasks, Crews, and the Process Abstraction

You've probably managed a project before — even informally. You know who to ask about what, what a good deliverable looks like, and roughly what order things need to happen. CrewAI formalises exactly that intuition into code.

The core idea: you describe your team (who they are, what they want, what they know) and what needs to be done. CrewAI works out how to run it.

## The three building blocks

Every CrewAI system is built from three concepts:

- **Agent** — who is doing the work
- **Task** — what needs to be done
- **Crew** — how the team is run

That's it. Once you internalise these three, the rest of the API is just configuration.

### Agent: who

An agent represents a role on your team. You define it with three fields:

```python
from crewai import Agent
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(model="claude-haiku-4-5-20251001")

researcher = Agent(
    role="Senior Research Scientist",
    goal="Find precise, evidence-backed answers to technical questions",
    backstory=(
        "You hold a PhD in Computer Science and spent a decade "
        "reviewing papers at top ML conferences. You never cite "
        "a source you haven't verified."
    ),
    llm=llm,
    verbose=True,
)
```

These three fields — `role`, `goal`, `backstory` — become the agent's system prompt. Under the hood CrewAI concatenates them into something like:

```
You are a Senior Research Scientist.
Your goal is: Find precise, evidence-backed answers to technical questions.
Your backstory: You hold a PhD in Computer Science...
```

That system prompt is sent to the LLM at the start of every task the agent handles.

### Task: what

A task is a contract between you and an agent. It has two required fields:

```python
from crewai import Task

research_task = Task(
    description=(
        "Research the current state of quantum error correction. "
        "Focus on surface codes and their practical threshold requirements."
    ),
    expected_output=(
        "A numbered list of 5 key findings, each with a one-paragraph "
        "explanation and at least one specific data point or citation."
    ),
    agent=researcher,
)
```

The `description` tells the agent what to do. The `expected_output` tells the agent what good looks like — and in hierarchical mode, the manager validates the agent's response against it before accepting it.

Think of `expected_output` as your acceptance criteria. Vague acceptance criteria produce vague outputs.

### Crew: how

The Crew assembles agents and tasks and specifies the process — the order and coordination strategy.

```python
from crewai import Crew, Process

crew = Crew(
    agents=[researcher],
    tasks=[research_task],
    process=Process.sequential,
    verbose=True,
)

result = crew.kickoff()
print(result.raw)
```

`crew.kickoff()` is the single call that runs your entire pipeline.

## The role / goal / backstory trio in depth

These three fields shape behaviour more than any other configuration. They are not cosmetic labels — they are the agent's identity.

**`role`** is used as a short identifier. It appears in verbose output, in delegation messages, and in the manager's routing decisions during hierarchical runs. Keep it specific: "Senior Research Scientist" is better than "Researcher" because it signals seniority and scope to other agents that might delegate to it.

**`goal`** is the agent's optimisation target. It should be one sentence describing what success looks like from the agent's perspective. The agent will implicitly bias its outputs toward satisfying this goal, even when the task description pulls in a slightly different direction. If these conflict, goal usually wins.

**`backstory`** is expertise context. It makes the agent's behaviour credible by giving the LLM a persona to inhabit. A researcher with a "PhD in quantum physics" writes differently about quantum topics than one with no backstory — the LLM draws on that persona when choosing vocabulary, depth, and caveats.

<Callout type="insight">
The `backstory` field is not decoration — invest time in it. A researcher whose backstory says "You are known for citing only peer-reviewed sources and flagging when evidence is weak" will produce outputs that reflect exactly those traits. A vague backstory produces generic outputs. Two or three specific sentences describing expertise, working style, and known limitations are worth more than a paragraph of fluff.
</Callout>

## Tasks as contracts: description and expected_output

The `description` and `expected_output` fields work as a pair:

- `description` drives the agent's reasoning — it's the instruction
- `expected_output` drives the agent's formatting — it's the acceptance criterion

When you write `expected_output`, think in format terms: bullet list vs numbered list, word count, whether citations are required, whether a structured JSON is expected. The agent will try to match what you describe.

```python
analysis_task = Task(
    description="Analyse the quantum error correction research for practical barriers to near-term implementation.",
    expected_output=(
        "A structured report with three sections: "
        "(1) Top 3 barriers with severity ratings (High/Medium/Low), "
        "(2) Estimated timeline to overcome each, "
        "(3) One open question per barrier that future research should address."
    ),
    agent=analyst,
)
```

Concrete format instructions in `expected_output` dramatically improve output consistency across runs.

## Task context: passing outputs between tasks

Tasks can declare dependencies using the `context` parameter:

```python
summary_task = Task(
    description="Summarise the analysis into an executive briefing.",
    expected_output="A 3-paragraph executive summary suitable for a non-technical VP.",
    agent=writer,
    context=[research_task, analysis_task],
)
```

When `summary_task` runs, the outputs of `research_task` and `analysis_task` are automatically injected into the writer's context. The writer sees what the researcher found and what the analyst concluded — without you having to wire any of it manually.

You can chain as many context dependencies as you need. In sequential process, this is how information flows down the pipeline.

## The Process abstraction: sequential vs hierarchical

`Process` is the single most important design decision in a CrewAI system.

### Process.sequential

Tasks run in the order they appear in the `tasks` list. Each task gets context from tasks listed in its `context=[]`. There is no manager — tasks just run one after another.

```
task_1 → task_2 → task_3 → done
```

Sequential is predictable, cheap, and appropriate for well-defined pipelines where every step is known in advance.

### Process.hierarchical

CrewAI creates a manager agent (using `manager_llm` or a custom `manager_agent`) that orchestrates the work:

1. The manager reads all tasks and decides what to do first
2. It assigns a task to a worker agent
3. When the worker responds, the manager checks the output against `expected_output`
4. If unsatisfied, the manager reassigns — to the same or a different worker
5. When all tasks are satisfied, the manager synthesises the final answer

```
manager → assign → worker_A → validate → OK
manager → assign → worker_B → validate → not OK → reassign → worker_B → validate → OK
manager → synthesise → done
```

Hierarchical is adaptive, has implicit quality control, and costs 30–60% more tokens due to the manager's routing calls.

```python
from crewai import Crew, Process
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(model="claude-haiku-4-5-20251001")

crew = Crew(
    agents=[researcher, analyst, writer],
    tasks=[research_task, analysis_task, summary_task],
    process=Process.hierarchical,
    manager_llm=llm,
    verbose=True,
)
```

## How CrewAI compares to LangGraph and AutoGen

Understanding where CrewAI sits helps you choose the right tool for the job.

| Dimension | LangGraph | AutoGen | CrewAI |
|---|---|---|---|
| You define | A graph of nodes | A conversation thread | A team and tasks |
| Orchestration | Explicit edges | Conversation routing | Auto-inferred by Process |
| Flexibility | Very high | High | Moderate |
| Onboarding time | High | Medium | Low |
| Hierarchical QA | Manual | Manual | Built-in |

**LangGraph** gives you complete control — you wire every edge, every condition, every state transition. That's powerful but verbose. CrewAI infers the execution graph from your task list and `context` declarations.

**AutoGen** organises agents into a conversation thread. Messages flow between agents and you control turn-taking. CrewAI uses a structured task pipeline — there's no open conversation, just task → output → next task.

The right tool depends on how well-defined your pipeline is. If the structure is clear and stable, CrewAI's abstraction saves significant time. If you need fine-grained control over routing logic or need resumability, LangGraph is a better fit.

<Callout type="note">
CrewAI does not expose the underlying execution graph to you directly. If you need to inspect or modify how the orchestration works at a low level — for debugging, auditing, or custom routing — you will need to switch to LangGraph or add a Flow (covered in Lesson 5). CrewAI trades transparency for ease of use.
</Callout>
