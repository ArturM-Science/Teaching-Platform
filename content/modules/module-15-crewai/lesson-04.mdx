# Hierarchical Process: Manager Agent, Delegation, and Token Overhead

Sequential mode is predictable and cheap. But sometimes you need a pipeline that can catch bad outputs, reassign work, and adapt ├втВмтАЭ without you hardcoding every retry condition. That's what `Process.hierarchical` provides: an automatic manager that validates each worker's output and replans when something isn't good enough.

The catch is cost. The manager's routing calls are not free. This lesson teaches you exactly how hierarchical works, when it's worth the overhead, and how to measure the difference.

## How Process.hierarchical works

When you set `process=Process.hierarchical`, CrewAI creates a manager agent before the crew runs. The manager reads your task list and orchestrates the workers in a five-step loop:

1. **Plan** ├втВмтАЭ the manager reads all tasks and decides what to assign first and to whom
2. **Assign** ├втВмтАЭ the manager delegates a task to a worker agent
3. **Execute** ├втВмтАЭ the worker runs the task using its tools and LLM
4. **Validate** ├втВмтАЭ the manager compares the worker's output against the task's `expected_output`
5. **Accept or reassign** ├втВмтАЭ if the output is satisfactory, the manager moves to the next task; if not, it reassigns (to the same or a different worker) and loops from step 2

```
manager ├втАЭтВм├втАЭтВмplan├втАЭтВм├втАЭтВм├втАУ┬║ task_1 ├втАЭтВм├втАЭтВмassign├втАЭтВм├втАЭтВм├втАУ┬║ researcher
                                         ├втАЭтАЪ
                          ├втАФтАЮ├втАЭтВм├втАЭтВмoutput├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭ╦Ь
                          ├втАЭтАЪ
                    validate vs expected_output
                          ├втАЭтАЪ
                   OK? ├втАЭтВм├втАЭтВм├втАЭтВм├втАЭ┬┤├втАЭтВм├втАЭтВм├втАУ┬║ task_2 ├втАЭтВм├втАЭтВмassign├втАЭтВм├втАЭтВм├втАУ┬║ analyst
                   NO? ├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАУ┬║ reassign ├втАЭтВм├втАЭтВм├втАУ┬║ researcher
```

When all tasks are validated, the manager synthesises the final answer.

## Configuring hierarchical

The minimum configuration: provide `manager_llm`.

```python
from crewai import Agent, Task, Crew, Process
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(model="claude-haiku-4-5-20251001")

researcher = Agent(
    role="Senior Researcher",
    goal="Find precise, evidence-backed answers",
    backstory="PhD in CS, 10 years reviewing papers at top ML conferences.",
    llm=llm,
    allow_delegation=False,
    verbose=True,
)

analyst = Agent(
    role="Critical Analyst",
    goal="Identify risks and practical implications",
    backstory="Specialist in translating research into actionable insights.",
    llm=llm,
    allow_delegation=False,
    verbose=True,
)

research_task = Task(
    description="Research {topic} and identify 5 key developments.",
    expected_output=(
        "A numbered list of 5 developments. Each must include a name, "
        "a 2-sentence explanation, and a specific data point or citation."
    ),
    agent=researcher,
)

analysis_task = Task(
    description="Analyse the research and identify the top 3 risks.",
    expected_output=(
        "A structured analysis with 3 numbered risks. "
        "Each risk must have a title, severity rating (High/Medium/Low), "
        "and a 2-sentence practical implication."
    ),
    agent=analyst,
    context=[research_task],
)

# ├втАЭтВм├втАЭтВм Hierarchical crew ├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм
crew = Crew(
    agents=[researcher, analyst],
    tasks=[research_task, analysis_task],
    process=Process.hierarchical,
    manager_llm=llm,          # CrewAI creates the manager from this LLM
    verbose=True,
)

result = crew.kickoff(inputs={"topic": "AI inference optimisation"})
print(result.raw)
print(crew.usage_metrics)
```

## Using a custom manager agent

Instead of letting CrewAI auto-create the manager, you can define one yourself. This gives the manager its own role, goal, backstory ├втВмтАЭ and its own tools:

```python
manager = Agent(
    role="Project Manager",
    goal=(
        "Deliver high-quality, accurate outputs by delegating effectively "
        "and enforcing strict quality standards."
    ),
    backstory=(
        "You have managed technical research teams for 15 years. "
        "You are known for catching errors before they propagate and "
        "for giving precise, actionable feedback when reassigning work."
    ),
    llm=llm,
    allow_delegation=True,    # manager must delegate ├втВмтАЭ always True for manager
    verbose=True,
)

crew = Crew(
    agents=[researcher, analyst],
    tasks=[research_task, analysis_task],
    process=Process.hierarchical,
    manager_agent=manager,    # use this instead of manager_llm
    verbose=True,
)
```

Use a custom manager when:
- You want the manager to have tools (e.g., a quality-check tool that scores outputs)
- You want the manager's reasoning to reflect a specific domain or style
- You're debugging and need to inspect the manager's full system prompt

When you set `manager_agent`, do not also set `manager_llm` ├втВмтАЭ they are mutually exclusive.

<Callout type="insight">
The hierarchical process's real value is implicit quality control. In sequential mode, a bad output silently propagates ├втВмтАЭ the analyst takes bad research at face value, the writer takes bad analysis at face value. In hierarchical mode, the manager validates each output against `expected_output` before accepting it. One well-written `expected_output` description can prevent an entire class of downstream errors. This is why tight `expected_output` strings matter even more in hierarchical mode than in sequential.
</Callout>

## allow_delegation on workers

By default, worker agents have `allow_delegation=False`. This means a worker cannot ask another worker for help ├втВмтАЭ it can only use its own tools and LLM.

Set `allow_delegation=True` on a worker when you want the manager to be able to ask it to consult another worker:

```python
senior_analyst = Agent(
    role="Senior Analyst",
    goal="Provide rigorous analysis, consulting specialists when needed",
    backstory="You lead analysis and know when to bring in domain experts.",
    llm=llm,
    allow_delegation=True,   # can ask other agents for input
)

junior_researcher = Agent(
    role="Domain Specialist",
    goal="Answer domain-specific technical questions with precision",
    backstory="PhD in quantum computing. You answer specific technical questions.",
    llm=llm,
    allow_delegation=False,  # focused executor ├втВмтАЭ does not delegate further
)
```

In practice, `allow_delegation=True` on workers adds complexity and cost. Use it only when your tasks genuinely require inter-agent consultation ├втВмтАЭ for example, when an analyst might need to ask a specialist a targeted question before completing their output.

## Measuring the token overhead

Run the same pipeline as both sequential and hierarchical and compare `usage_metrics`:

```python
import os
from crewai import Agent, Task, Crew, Process
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(model="claude-haiku-4-5-20251001")

def make_agents():
    researcher = Agent(
        role="Senior Researcher",
        goal="Find precise answers",
        backstory="10 years in technical research.",
        llm=llm, allow_delegation=False, verbose=False,
    )
    analyst = Agent(
        role="Critical Analyst",
        goal="Identify risks and implications",
        backstory="Expert in translating research to action.",
        llm=llm, allow_delegation=False, verbose=False,
    )
    return researcher, analyst

def make_tasks(researcher, analyst):
    research_task = Task(
        description="Research {topic} and identify 5 key developments.",
        expected_output=(
            "A numbered list of 5 developments, each with a name, "
            "2-sentence explanation, and one data point."
        ),
        agent=researcher,
    )
    analysis_task = Task(
        description="Analyse the research and list the top 3 risks.",
        expected_output=(
            "3 numbered risks, each with title, severity (High/Medium/Low), "
            "and 2-sentence implication."
        ),
        agent=analyst,
        context=[research_task],
    )
    return research_task, analysis_task

inputs = {"topic": "AI inference optimisation"}

# ├втАЭтВм├втАЭтВм Sequential run ├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм
r1, a1 = make_agents()
t1, t2 = make_tasks(r1, a1)
seq_crew = Crew(agents=[r1, a1], tasks=[t1, t2], process=Process.sequential, verbose=False)
seq_crew.kickoff(inputs=inputs)
seq_metrics = seq_crew.usage_metrics

# ├втАЭтВм├втАЭтВм Hierarchical run ├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм
r2, a2 = make_agents()
t3, t4 = make_tasks(r2, a2)
hier_crew = Crew(
    agents=[r2, a2], tasks=[t3, t4],
    process=Process.hierarchical, manager_llm=llm, verbose=False,
)
hier_crew.kickoff(inputs=inputs)
hier_metrics = hier_crew.usage_metrics

# ├втАЭтВм├втАЭтВм Compare ├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм├втАЭтВм
print(f"Sequential  ├втВмтАЭ total tokens: {seq_metrics.total_tokens:,}  |  requests: {seq_metrics.successful_requests}")
print(f"Hierarchical ├втВмтАЭ total tokens: {hier_metrics.total_tokens:,}  |  requests: {hier_metrics.successful_requests}")
overhead = (hier_metrics.total_tokens - seq_metrics.total_tokens) / seq_metrics.total_tokens * 100
print(f"Overhead: {overhead:.1f}%")
```

Typical output:

```
Sequential   ├втВмтАЭ total tokens: 3,241  |  requests: 2
Hierarchical ├втВмтАЭ total tokens: 4,892  |  requests: 5
Overhead: 51.0%
```

The extra requests in hierarchical output are the manager's planning, validation, and synthesis calls. Those calls are the dominant cost ├втВмтАЭ not the workers' extra reasoning.

<Callout type="note">
The overhead percentage varies by task. Simple tasks with clear `expected_output` descriptions produce less overhead because the manager validates and accepts quickly ├втВмтАЭ fewer reassignment loops. Complex tasks with vague acceptance criteria can produce 80├втВмтАЬ100% overhead as the manager loops multiple times before accepting an output. Writing tight `expected_output` strings reduces hierarchical cost, sometimes dramatically.
</Callout>

## When to use hierarchical vs sequential

Use **Process.sequential** when:
- Your pipeline is well-defined and all steps are known in advance
- Each step's `expected_output` is clear and the agent reliably meets it
- Token cost is a primary constraint
- You want predictable, auditable execution where every step runs exactly once

Use **Process.hierarchical** when:
- Tasks require quality gates ├втВмтАЭ you need a check before output propagates
- Outputs are hard to specify precisely and the manager's judgment adds value
- Your pipeline involves replanning (a failed step might need a different approach)
- You want dynamic reassignment without writing retry logic yourself

Use **neither** ├втВмтАЭ use LangGraph ├втВмтАЭ when:
- You need checkpointing or resumability (the crew can be interrupted and restarted)
- You need fine-grained control over routing logic
- You need to inspect or modify the execution graph at runtime

## A practical decision heuristic

Answer these questions in order:

1. Can I write a precise `expected_output` for every task? ├втАатАЩ If yes, sequential is likely sufficient.
2. Will a bad output from task N cascade into task N+1 in a way that's hard to detect? ├втАатАЩ If yes, hierarchical's validation loop helps.
3. Do I have a token budget constraint? ├втАатАЩ If tight, sequential only.
4. Do my tasks require dynamic reassignment based on content? ├втАатАЩ If yes, hierarchical. If the condition is complex, use a Flow.
